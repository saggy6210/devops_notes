Kubernetes Architecture Questions

1

How would you design a multi-cluster Kubernetes strategy for a global enterprise?

Key considerations:

  • Cluster topology: Regional clusters for latency, central management cluster
  • Workload placement: Data residency requirements, user proximity
  • Federation options: Kubefed, Liqo, or GitOps-based (ArgoCD ApplicationSets)
  • Service mesh: Cross-cluster communication (Istio multi-cluster, Linkerd)
  • DNS strategy: Global load balancing (Route53, CloudFlare)

Management approaches:

  • Fleet management: Rancher, OpenShift ACM, Tanzu Mission Control
  • GitOps: Single source of truth for all clusters
  • Observability: Centralized monitoring across clusters (Thanos, Cortex)
2

How do you implement multi-tenancy in Kubernetes? What are the isolation levels?

Isolation levels (soft to hard):

  • Namespace isolation: RBAC, Network Policies, Resource Quotas
  • Virtual clusters: vCluster - virtual control planes in namespaces
  • Cluster per tenant: Full isolation but higher cost

Key controls for namespace multi-tenancy:

  • RBAC: Tenant-scoped roles, no cluster-admin
  • Network Policies: Default deny, explicit allow between namespaces
  • Resource Quotas: CPU, memory, storage limits per namespace
  • LimitRanges: Default and max resource limits for pods
  • Pod Security Standards: Restricted profile for tenants
  • OPA/Kyverno: Policy enforcement (no privileged pods, required labels)

Tools: Capsule, Hierarchical Namespaces Controller (HNC), Loft

3

How would you design etcd for a production Kubernetes cluster?

Sizing and topology:

  • Cluster size: 3 or 5 nodes (odd number for quorum)
  • Dedicated nodes: Separate from control plane for large clusters
  • Resources: SSD storage (critical), low-latency network between nodes

High availability:

  • Spread across availability zones
  • Use anti-affinity rules
  • Quorum: 3 nodes tolerate 1 failure, 5 tolerate 2

Backup strategy:

  • Regular snapshots (etcdctl snapshot save)
  • Store backups off-cluster (S3, GCS)
  • Test restore procedures regularly
  • Velero for cluster-level backup

Performance tuning: Increase heartbeat interval for cross-AZ, monitor latency metrics.

4

How do you design Kubernetes networking for enterprise? CNI selection criteria?

CNI selection criteria:

  • Network Policy support: Essential for security (Calico, Cilium)
  • Performance: eBPF-based for high throughput (Cilium)
  • Encryption: WireGuard integration (Calico, Cilium)
  • Observability: Network flow logs, metrics
  • Multi-cluster: Cross-cluster connectivity

Enterprise considerations:

  • IP address management: Integration with corporate IPAM
  • BGP peering: Direct integration with network infrastructure
  • Egress controls: Restrict outbound traffic, NAT gateways
  • Service mesh integration: Sidecar or eBPF-based (Cilium Service Mesh)

Popular choices: Calico (enterprise features), Cilium (eBPF, observability), AWS VPC CNI, Azure CNI

5

How do you handle stateful workloads in Kubernetes at scale?

StatefulSet best practices:

  • Stable network identities (headless service)
  • Ordered deployment and scaling
  • PodDisruptionBudgets for maintenance
  • Anti-affinity for HA

Storage considerations:

  • Storage classes: SSD for databases, appropriate IOPS
  • CSI drivers: Cloud-native (EBS CSI, Azure Disk) or enterprise (NetApp Trident, Portworx)
  • Volume expansion: Enable allowVolumeExpansion
  • Snapshots: VolumeSnapshots for backups

Operators for databases:

  • PostgreSQL: CloudNativePG, Zalando Operator
  • MySQL: Oracle MySQL Operator, Percona
  • MongoDB: MongoDB Community Operator
  • Redis: Redis Operator, Spotahome

Alternative: Use managed databases outside K8s for critical workloads.

6

How do you design Kubernetes for zero-downtime deployments?

Deployment strategies:

  • Rolling update: maxSurge and maxUnavailable tuning
  • Blue-green: Service selector switch
  • Canary: Flagger, Argo Rollouts with traffic splitting

Application requirements:

  • Readiness probes: Only receive traffic when ready
  • Liveness probes: Restart unhealthy pods
  • Startup probes: For slow-starting applications
  • PreStop hooks: Graceful shutdown, drain connections
  • terminationGracePeriodSeconds: Sufficient time for cleanup

Infrastructure requirements:

  • PodDisruptionBudgets to limit simultaneous pod evictions
  • Pod anti-affinity across nodes/zones
  • Ingress connection draining
7

How do you implement GitOps for Kubernetes at enterprise scale?

Repository structure:

  • App repo: Application code and Dockerfile
  • Config repo: Kubernetes manifests, Helm charts
  • Environment overlays: Kustomize for dev/staging/prod

Tools comparison:

  • ArgoCD: UI, RBAC, multi-cluster, ApplicationSets for fleet
  • Flux: Lightweight, GitOps Toolkit, better Helm integration

Enterprise patterns:

  • ApplicationSets for multi-cluster deployment
  • App-of-apps pattern for hierarchy
  • Sync waves for ordered deployment
  • Progressive delivery with Argo Rollouts
  • Secrets: Sealed Secrets, External Secrets, SOPS

Governance: PR-based changes, required reviews, automated testing in CI.

8

How do you secure a Kubernetes cluster for production?

Control plane security:

  • API server: Enable audit logging, restrict anonymous access
  • etcd: Encrypt at rest, mTLS between nodes
  • Authentication: OIDC integration (no static tokens)
  • Private API endpoint or IP allowlisting

Workload security:

  • Pod Security Standards (restricted)
  • Network Policies (default deny)
  • RBAC with least privilege
  • Service account token projection (bound tokens)
  • Secrets encryption at rest

Supply chain:

  • Image signing and verification (Cosign, Sigstore)
  • Admission controllers for image policies
  • Private registry with vulnerability scanning

Runtime:

  • Falco for runtime threat detection
  • Seccomp and AppArmor profiles
9

How do you design Kubernetes autoscaling for varying workloads?

Pod-level scaling:

  • HPA (Horizontal Pod Autoscaler): Scale based on CPU, memory, or custom metrics
  • VPA (Vertical Pod Autoscaler): Right-size resource requests
  • KEDA: Event-driven scaling (queue depth, cron, external metrics)

Node-level scaling:

  • Cluster Autoscaler: Add/remove nodes based on pending pods
  • Karpenter: Faster provisioning, better bin-packing, spot instances

Best practices:

  • Set accurate resource requests (basis for HPA)
  • Use custom metrics for business-relevant scaling
  • Scale-to-zero with KEDA for batch workloads
  • Pod priority and preemption for critical workloads
  • Buffer capacity for faster scaling
10

How do you handle secrets management in Kubernetes at scale?

Problems with native secrets:

  • Base64 encoded, not encrypted in etcd by default
  • Anyone with namespace access can read
  • Secrets in Git are a security risk

Solutions:

  • External Secrets Operator: Sync from Vault, AWS Secrets Manager, Azure Key Vault
  • Sealed Secrets: Encrypt secrets for Git storage
  • HashiCorp Vault: Dynamic secrets, PKI, audit logging
  • SOPS + Age/GPG: Encrypt secrets files in Git

Best practices:

  • Enable etcd encryption at rest
  • Use short-lived credentials where possible
  • Rotate secrets automatically
  • Audit secret access via K8s audit logs
  • Workload identity (IRSA, Workload Identity) instead of static credentials
11

How do you design ingress and API gateway for Kubernetes?

Ingress controller selection:

  • NGINX Ingress: Most common, good for standard use cases
  • Traefik: Native K8s integration, auto-discovery
  • HAProxy: High performance, enterprise features
  • Kong: API gateway features built-in
  • Envoy-based: Contour, Emissary (Ambassador)

Gateway API (next-gen):

  • More expressive than Ingress
  • Role-oriented: Infrastructure provider, cluster operator, app developer
  • Traffic splitting, header-based routing native

Enterprise patterns:

  • External load balancer → Ingress controller → Services
  • TLS termination at ingress or end-to-end
  • Rate limiting, WAF integration
  • Multi-tenant ingress with IngressClass
12

How do you implement disaster recovery for Kubernetes?

What to backup:

  • etcd: Cluster state (critical)
  • Persistent volumes: Application data
  • Kubernetes resources: Deployments, ConfigMaps, Secrets

Tools:

  • Velero: Backup resources and PVs, cross-cluster restore
  • Kasten K10: Enterprise backup, application-aware
  • etcd snapshots: For control plane recovery

DR strategies:

  • Active-Passive: Standby cluster, restore from backup
  • Active-Active: Multi-cluster with traffic failover
  • GitOps recovery: Rebuild cluster from Git (stateless only)

Testing: Regular DR drills, document RTO/RPO, automate recovery procedures.

13

How do you optimize Kubernetes costs?

Right-sizing:

  • VPA recommendations for resource requests
  • Goldilocks for namespace-wide analysis
  • Avoid over-provisioning (requests too high)

Efficient scheduling:

  • Karpenter: Better bin-packing than Cluster Autoscaler
  • Pod priority: Evict low-priority for critical workloads
  • Spot/Preemptible instances: For fault-tolerant workloads

Visibility:

  • Kubecost: Cost allocation by namespace, label
  • OpenCost: CNCF project for cost monitoring
  • Chargeback to teams based on usage

Operational:

  • Scale down dev/staging outside hours
  • Reserved instances for baseline, spot for burst
  • Clean up unused resources (PVs, load balancers)
14

How do you upgrade Kubernetes clusters with minimal disruption?

Upgrade strategy:

  • One minor version at a time (1.27 → 1.28 → 1.29)
  • Read release notes for deprecations and breaking changes
  • Test in non-production first

Control plane upgrade:

  • Managed K8s: Usually handled by provider with zero downtime
  • Self-managed: Upgrade one master at a time, etcd backup first

Node upgrade (rolling):

  1. Cordon node (prevent new scheduling)
  2. Drain node (evict pods with PDB respect)
  3. Upgrade kubelet and container runtime
  4. Uncordon node

Blue-green nodes: Add new node pool with new version, migrate workloads, remove old pool.

Pre-upgrade checks: Deprecated APIs (pluto, kubent), PDBs configured, pod resource limits set.

How do you implement service mesh in Kubernetes? When is it needed?

When to use service mesh:

  • Many microservices with complex traffic patterns
  • Need mTLS without application changes
  • Advanced traffic management (canary, fault injection)
  • Consistent observability across services

When NOT to use:

  • Simple applications (few services)
  • Team lacks expertise to operate
  • Latency-sensitive applications (sidecar overhead)

Options:

  • Istio: Feature-rich, complex, high resource usage
  • Linkerd: Lightweight, simpler, lower overhead
  • Cilium: eBPF-based, no sidecars, high performance
  • Consul Connect: Works outside K8s too

Implementation: Start with observability, add mTLS, then advanced traffic features.

How do you troubleshoot Kubernetes networking issues?

Common issues:

  • Pod-to-pod communication failing
  • Service not reachable
  • Ingress/external access issues
  • DNS resolution problems

Troubleshooting steps:

  1. Check endpoints: kubectl get endpoints <svc>
  2. DNS test: kubectl exec -it pod -- nslookup service-name
  3. Connectivity: kubectl exec -it pod -- curl service:port
  4. Network policies: Check if policies block traffic
  5. CNI logs: Check CNI plugin logs on nodes

Tools:

  • netshoot: Debug container with network tools
  • kubectl debug: Ephemeral debug containers
  • Hubble (Cilium): Network flow visibility
  • tcpdump/wireshark: Packet capture on nodes
17

How do you implement observability for Kubernetes?

Metrics:

  • Prometheus: De facto standard for K8s
  • kube-state-metrics: K8s object states
  • metrics-server: Resource metrics for HPA
  • Node exporter: Node-level metrics

Logging:

  • Stack: Fluentd/Fluent Bit → Elasticsearch/Loki → Kibana/Grafana
  • Cloud: CloudWatch, Stackdriver, Azure Monitor
  • Structured logging with correlation IDs

Tracing:

  • OpenTelemetry for instrumentation
  • Jaeger, Tempo for trace storage
  • Service mesh provides automatic tracing

Dashboards: Grafana with pre-built K8s dashboards, custom SLO dashboards.

What's your approach to Kubernetes platform team structure?

Team responsibilities:

  • Platform team: Cluster lifecycle, upgrades, security baseline, shared services
  • Application teams: Deploy and operate their applications

Platform as a product:

  • Self-service for developers (namespaces, resources)
  • Golden paths and templates
  • Internal developer portal (Backstage)
  • Clear SLOs for platform reliability

Tooling provided:

  • CI/CD pipelines and templates
  • Observability stack
  • Secrets management
  • Ingress and DNS automation

Documentation: Runbooks, architecture decision records (ADRs), onboarding guides.
