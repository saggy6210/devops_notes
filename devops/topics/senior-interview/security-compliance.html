Security & Compliance Questions

1

What is DevSecOps? How is it different from DevOps?

DevSecOps integrates security practices into every phase of the DevOps lifecycle.

Key principles:

  • Security as code: Security policies defined and enforced through code
  • Shift left: Security testing early in development
  • Automation: Automated security scanning in CI/CD
  • Shared responsibility: Everyone owns security, not just security team

Difference: DevOps focuses on speed and collaboration; DevSecOps adds security without sacrificing velocity.

2

What is Zero Trust Architecture?

Zero Trust is a security model based on "never trust, always verify" - no implicit trust based on network location.

Principles:

  • Verify explicitly: Authenticate and authorize every request
  • Least privilege: Minimum access required for the task
  • Assume breach: Segment access, verify end-to-end

Implementation:

  • Identity-based access (not network-based)
  • Micro-segmentation
  • mTLS between services
  • Continuous verification
3

How do you manage secrets in a DevOps environment?

Never store secrets in: Source code, environment variables (visible in process list), config files in repos.

Solutions:

  • HashiCorp Vault: Dynamic secrets, encryption as a service, audit logging
  • Cloud providers: AWS Secrets Manager, Azure Key Vault, GCP Secret Manager
  • Kubernetes: Sealed Secrets, External Secrets Operator

Best practices:

  • Rotate secrets regularly
  • Use short-lived credentials when possible
  • Audit access to secrets
  • Encrypt secrets at rest and in transit
4

What security scanning should be in a CI/CD pipeline?

  • SAST (Static Application Security Testing): Analyze source code for vulnerabilities (SonarQube, Checkmarx, Semgrep)
  • SCA (Software Composition Analysis): Scan dependencies for known CVEs (Snyk, Dependabot, OWASP Dependency-Check)
  • Secret scanning: Detect committed secrets (GitLeaks, TruffleHog)
  • Container scanning: Image vulnerabilities (Trivy, Clair, Anchore)
  • IaC scanning: Terraform/CloudFormation misconfigurations (Checkov, tfsec)
  • DAST (Dynamic): Runtime testing in staging (OWASP ZAP)
5

How do you secure container images?

Build time:

  • Use minimal base images (distroless, Alpine)
  • Don't run as root
  • Multi-stage builds to reduce attack surface
  • Scan images for vulnerabilities
  • Pin base image versions (avoid :latest)

Registry:

  • Private registry with authentication
  • Image signing (Cosign, Notary)
  • Admission controllers to enforce signed images

Runtime:

  • Read-only file systems
  • Drop capabilities
  • Network policies
6

What is Pod Security in Kubernetes?

Pod Security Standards (replaced PodSecurityPolicies):

  • Privileged: Unrestricted (for system components)
  • Baseline: Minimally restrictive, prevents known privilege escalations
  • Restricted: Heavily restricted, security best practices

Key controls:

  • runAsNonRoot: true
  • readOnlyRootFilesystem: true
  • allowPrivilegeEscalation: false
  • Drop all capabilities, add only needed
  • seccompProfile: RuntimeDefault

Alternatives: OPA Gatekeeper, Kyverno for custom policies.

7

Explain RBAC in Kubernetes. Best practices?

RBAC (Role-Based Access Control) controls who can do what in Kubernetes.

Components:

  • Role/ClusterRole: Define permissions (verbs on resources)
  • RoleBinding/ClusterRoleBinding: Grant roles to users/groups/service accounts

Best practices:

  • Least privilege: Start with minimal permissions
  • Namespace isolation: Use Roles over ClusterRoles
  • Avoid wildcard permissions (*)
  • Regular audit: kubectl auth can-i
  • Group-based access: Bind to groups, not users
  • Restrict cluster-admin: Only for emergencies
8

What is mTLS? When should you use it?

mTLS (Mutual TLS) authenticates both client and server, not just server (standard TLS).

How it works:

  1. Server presents certificate to client
  2. Client verifies server certificate
  3. Client presents certificate to server
  4. Server verifies client certificate

Use cases:

  • Service-to-service communication in microservices
  • Zero Trust networks
  • API authentication

Implementation: Service mesh (Istio, Linkerd) handles mTLS automatically.

9

How do you implement network segmentation in Kubernetes?

Network Policies: Define allowed traffic between pods.

Default deny example:

  • Deny all ingress by default
  • Explicitly allow needed traffic
  • Label-based selection

Best practices:

  • Start with deny-all policy
  • Namespace isolation
  • Separate environments (dev/staging/prod)
  • Restrict egress to known destinations

Requirements: CNI that supports Network Policies (Calico, Cilium, Weave Net).

10

What is supply chain security? SLSA framework?

Supply chain security protects the entire software delivery pipeline from source to deployment.

Threats: Compromised dependencies, tampered builds, malicious insiders.

SLSA (Supply chain Levels for Software Artifacts):

  • Level 1: Documentation of build process
  • Level 2: Tamper resistance (signed provenance)
  • Level 3: Hardened builds (isolated, reproducible)
  • Level 4: Two-person review, hermetic builds

Tools: Sigstore (Cosign, Fulcio, Rekor), in-toto, SBOM generators.

11

What is an SBOM? Why is it important?

SBOM (Software Bill of Materials) is a complete inventory of all components in software.

Contents:

  • Direct and transitive dependencies
  • Version information
  • Licenses
  • Supplier information

Why important:

  • Vulnerability management (quickly identify affected systems)
  • License compliance
  • Regulatory requirements (US Executive Order)
  • Incident response

Formats: SPDX, CycloneDX. Tools: Syft, Trivy, FOSSA.

12

How do you handle compliance in cloud-native environments?

Policy as Code:

  • OPA/Rego policies for Kubernetes
  • Cloud security posture management (CSPM)
  • Infrastructure compliance scanning

Audit and logging:

  • Kubernetes audit logs
  • Cloud trail/activity logs
  • Immutable audit storage

Continuous compliance:

  • Automated compliance checks in CI/CD
  • Real-time drift detection
  • Automated remediation

Frameworks: CIS Benchmarks, SOC 2, PCI-DSS, HIPAA, GDPR.

13

What is runtime security for containers?

Runtime security monitors and protects containers during execution.

Capabilities:

  • Behavioral monitoring: Detect anomalous activity
  • System call filtering: Seccomp profiles
  • File integrity: Detect unexpected changes
  • Network monitoring: Unexpected connections
  • Process monitoring: Unexpected processes

Tools: Falco (CNCF), Sysdig, Aqua, Prisma Cloud.

Example threats detected: Shell in container, crypto mining, privilege escalation attempts.

14

How do you secure the CI/CD pipeline itself?

Pipeline security measures:

  • Access control: RBAC for pipeline modifications
  • Secrets: External secrets management, no hardcoded credentials
  • Ephemeral runners: Fresh environment each build
  • Signed commits: GPG signing, branch protection
  • Artifact signing: Sign build outputs
  • Isolated builds: Sandboxed build environments

Additional controls:

  • Required reviewers for pipeline changes
  • Audit logging for all pipeline activities
  • Network isolation for build agents
15

What is the principle of least privilege? How do you implement it?

Least privilege: Grant only the minimum access required to perform a task.

Implementation:

  • IAM: Specific policies, not admin access
  • Service accounts: Dedicated accounts with minimal permissions
  • Just-in-time access: Temporary elevated permissions
  • Network: Restrict to needed ports/destinations
  • Containers: Non-root, dropped capabilities
  • Databases: Read-only accounts where possible

Audit: Regular access reviews, remove unused permissions.
