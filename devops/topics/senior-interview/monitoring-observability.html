Monitoring & Observability Questions

1

What is Observability? How is it different from Monitoring?

Monitoring: Collecting predefined metrics to detect known failure modes. "Is the system working?"

Observability: Ability to understand internal system state from external outputs. "Why is the system behaving this way?"

Three Pillars of Observability:

  • Metrics: Numeric measurements over time (CPU, latency, error rates)
  • Logs: Discrete events with context
  • Traces: Request flow across services

Key insight: Monitoring tells you something is wrong; observability helps you figure out why.

2

Explain SLI, SLO, and SLA. How do they relate?

SLI (Service Level Indicator): A measurable metric (e.g., request latency, availability).

SLO (Service Level Objective): Target value for an SLI (e.g., 99.9% availability, p99 latency < 200ms).

SLA (Service Level Agreement): Contract with consequences if SLOs aren't met (e.g., refunds).

Relationship: SLIs measure → SLOs set targets → SLAs define consequences

Error Budget = 100% - SLO (e.g., 99.9% SLO = 0.1% error budget = ~43 min/month downtime allowed)

3

What are the four golden signals?

The Four Golden Signals (from Google SRE book) are the most important metrics for monitoring services:

  • Latency: Time to serve a request (distinguish successful vs. failed requests)
  • Traffic: Demand on the system (requests/second, sessions, transactions)
  • Errors: Rate of failed requests (explicit, implicit, or policy-based)
  • Saturation: How "full" the system is (CPU, memory, queue depth)

Alternative: RED method (Rate, Errors, Duration) for microservices, USE method (Utilization, Saturation, Errors) for resources.

4

Explain Prometheus architecture. What are its components?

Core components:

  • Prometheus Server: Scrapes, stores, and queries metrics
  • TSDB: Time-series database for efficient storage
  • Exporters: Expose metrics from systems (node_exporter, blackbox_exporter)
  • Pushgateway: For short-lived jobs to push metrics
  • Alertmanager: Handles alerts, deduplication, routing, silencing
  • Service Discovery: Auto-discover targets (Kubernetes, Consul, DNS)

Data model: Metrics identified by name and key-value labels. E.g., http_requests_total{method="GET", status="200"}

5

What is the difference between Prometheus metric types?

  • Counter: Monotonically increasing value (requests served, errors). Use rate() to get per-second rate.
  • Gauge: Value that can go up and down (temperature, memory usage, queue size).
  • Histogram: Samples observations into configurable buckets. Use for latency distributions. Server-side quantile calculation.
  • Summary: Similar to histogram but calculates quantiles on client side. More accurate but less flexible.

Rule of thumb: Use histograms for latency unless you need precise quantiles and can't aggregate.

6

What is distributed tracing? How does it work?

Distributed tracing tracks requests as they flow through multiple services.

Concepts:

  • Trace: End-to-end journey of a request
  • Span: Single operation within a trace (with start time, duration, metadata)
  • Context propagation: Trace ID passed between services (via headers)

Standards: OpenTelemetry (OTLP), W3C Trace Context

Tools: Jaeger, Zipkin, Tempo, AWS X-Ray, Datadog APM

Use cases: Finding slow spans, understanding dependencies, debugging distributed failures.

7

What is OpenTelemetry? Why is it important?

OpenTelemetry (OTel) is a vendor-neutral standard for collecting telemetry data (traces, metrics, logs).

Components:

  • APIs & SDKs: Instrumentation libraries for multiple languages
  • Collector: Receives, processes, and exports telemetry data
  • OTLP: Protocol for transmitting telemetry

Benefits:

  • Vendor-agnostic: Switch backends without code changes
  • Unified approach: Single standard for all telemetry
  • CNCF graduated project: Wide adoption
8

How do you design effective alerting?

Principles:

  • Alert on symptoms, not causes: "Users can't check out" not "CPU is high"
  • Every alert should be actionable: If no action needed, it's not an alert
  • Reduce noise: Alert fatigue leads to ignored alerts

Severity levels:

  • Critical (P1): Revenue impact, wake people up
  • Warning (P2): Handle next business day
  • Info: Don't alert, just log

Features: Aggregation, deduplication, silencing, escalation policies.

9

What is log aggregation? Best practices?

Log aggregation collects logs from multiple sources into a centralized system.

Architecture: Collection (Fluentd, Filebeat) → Processing → Storage (Elasticsearch, Loki) → Visualization (Kibana, Grafana)

Best practices:

  • Structured logging: JSON format with consistent fields
  • Correlation IDs: Link logs to traces
  • Log levels: Use appropriately (ERROR, WARN, INFO, DEBUG)
  • Retention policies: Based on compliance and cost
  • Sampling: For high-volume services
  • Avoid PII: Mask sensitive data
10

How do you scale Prometheus for large environments?

Challenges: Single instance limitations, storage, query performance.

Solutions:

  • Federation: Hierarchical Prometheus servers
  • Thanos: Global query view, long-term storage in object storage
  • Cortex/Mimir: Horizontally scalable, multi-tenant
  • Victoria Metrics: High-performance alternative

Best practices:

  • Reduce cardinality (fewer label combinations)
  • Use recording rules for expensive queries
  • Appropriate scrape intervals
  • Shard by service/team
11

What is cardinality in monitoring? Why does it matter?

Cardinality is the number of unique time series (metric name + unique label combinations).

Why it matters:

  • High cardinality = more memory, storage, slower queries
  • Common cause of Prometheus OOM

High cardinality sources:

  • User IDs as labels
  • Request IDs or trace IDs
  • Full URLs with query params
  • Unbounded values as labels

Solutions: Use bounded labels, move high-cardinality data to logs/traces, relabeling to drop.

12

What is synthetic monitoring?

Synthetic monitoring (proactive monitoring) uses simulated requests to test system behavior.

Types:

  • Uptime checks: Simple HTTP/TCP probes
  • API tests: Validate endpoints and responses
  • Browser tests: Full page loads, user journeys
  • Multi-step transactions: Login → Search → Checkout

Tools: Prometheus Blackbox Exporter, Checkly, Datadog Synthetic, Pingdom

Benefits: Detect issues before users, test from multiple locations, measure SLIs.

13

How do you monitor Kubernetes clusters?

Cluster-level metrics:

  • kube-state-metrics: K8s object states (deployments, pods)
  • metrics-server: Resource usage for HPA
  • node-exporter: Node metrics (CPU, memory, disk)

Application-level:

  • ServiceMonitors for auto-discovery (Prometheus Operator)
  • Application metrics via annotations

Key things to monitor:

  • Pod restarts, OOMKilled events
  • Pending pods, failed scheduling
  • Node conditions, resource pressure
  • API server latency and errors
14

What is AIOps? How does it apply to observability?

AIOps (Artificial Intelligence for IT Operations) uses ML to enhance IT operations.

Applications in observability:

  • Anomaly detection: Identify unusual patterns without manual thresholds
  • Alert correlation: Group related alerts to reduce noise
  • Root cause analysis: Automatically identify probable causes
  • Predictive alerts: Warn before failures occur
  • Log analysis: Pattern detection, classification

Challenges: Requires quality data, can be a black box, false positives, model drift.
