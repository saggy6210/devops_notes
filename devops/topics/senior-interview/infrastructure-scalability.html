Infrastructure & Scalability Q&A

Master infrastructure questions covering scaling strategies, high availability, disaster recovery, and cloud architecture.

Infrastructure & Scalability Questions

1

Explain Vertical vs. Horizontal Scaling. When to use each?

Vertical Scaling (Scale Up):

  • Add more CPU, RAM, storage to existing machine
  • Simpler to implement, no code changes
  • Has hardware limits, single point of failure
  • Use for: Databases, legacy applications

Horizontal Scaling (Scale Out):

  • Add more machines to distribute load
  • Requires stateless design or distributed state
  • Virtually unlimited, better fault tolerance
  • Use for: Web servers, microservices, stateless APIs
2

What is the difference between High Availability and Disaster Recovery?

High Availability (HA):

  • Minimize downtime during normal operations
  • Redundancy within same region/datacenter
  • Automatic failover (seconds to minutes)
  • Metrics: 99.9% = 8.7h/year, 99.99% = 52m/year

Disaster Recovery (DR):

  • Recover from major failures (region outage, natural disaster)
  • Cross-region/cross-cloud replication
  • Defined by RTO (Recovery Time) and RPO (Recovery Point)
  • May require manual intervention
3

Explain Fault Tolerance vs. High Availability.

Fault Tolerance: System continues to operate with zero downtime even when components fail. No degradation in performance. Example: RAID arrays, redundant power supplies.

High Availability: System minimizes downtime but may have brief interruption during failover. May have degraded performance. Example: Active-passive database clusters.

Cost: Fault tolerance is more expensive (requires duplicate systems running simultaneously).

4

What are RTO and RPO? How do they affect architecture?

RTO (Recovery Time Objective): Maximum acceptable time to restore service after failure.

RPO (Recovery Point Objective): Maximum acceptable data loss measured in time (how old can the backup be?).

Architecture implications:

  • RTO = 0: Hot standby, active-active
  • RTO = minutes: Warm standby with automated failover
  • RPO = 0: Synchronous replication (affects latency)
  • RPO = hours: Async replication, regular backups
5

Explain Elasticity in cloud computing.

Elasticity is the ability to automatically scale resources up or down based on demand.

Components:

  • Auto-scaling: Add/remove instances based on metrics (CPU, memory, custom)
  • Predictive scaling: Scale based on forecasted demand
  • Scheduled scaling: Scale based on known patterns

Benefits: Cost optimization (pay for what you use), handle traffic spikes, maintain performance.

6

What is Resiliency? How do you build resilient systems?

Resiliency is the ability to recover from failures and continue functioning.

Patterns:

  • Circuit Breaker: Stop calling failing services
  • Retry with backoff: Exponential backoff for transient failures
  • Bulkhead: Isolate failures to prevent cascade
  • Timeout: Fail fast instead of waiting indefinitely
  • Graceful degradation: Reduce functionality instead of failing completely
  • Chaos engineering: Proactively test failure scenarios
7

Explain different load balancing algorithms.

  • Round Robin: Requests distributed sequentially. Simple but ignores server capacity.
  • Weighted Round Robin: Accounts for server capacity via weights.
  • Least Connections: Routes to server with fewest active connections.
  • Weighted Least Connections: Combines weights with connection count.
  • IP Hash: Same client IP always goes to same server (session affinity).
  • Resource Based: Routes based on server health/resource availability.
8

What are sticky sessions? Pros and cons?

Sticky sessions ensure requests from same client always go to same server.

Implementation: Cookie-based (application or duration-based) or IP-based.

Pros:

  • Simplifies stateful applications
  • Better cache utilization

Cons:

  • Uneven load distribution
  • Lost sessions on server failure
  • Harder to scale and maintain

Better approach: Externalized session storage (Redis)

9

What is connection draining? When is it important?

Connection draining (or deregistration delay) allows in-flight requests to complete before removing an instance from load balancer.

Process:

  1. Instance marked for removal
  2. No new connections sent to instance
  3. Existing connections allowed to complete (within timeout)
  4. Instance removed after timeout or all connections closed

Important for: Graceful deployments, auto-scaling down, instance maintenance.

10

How would you design a multi-region architecture?

Key considerations:

  • Traffic routing: GeoDNS or global load balancer (AWS Global Accelerator, CloudFlare)
  • Data replication: Active-active vs. active-passive, sync vs. async
  • Conflict resolution: Last-write-wins, CRDTs, or application logic
  • Latency: Keep user data close to users
  • Compliance: Data residency requirements

Patterns:

  • Active-Active: Both regions serve traffic, requires conflict handling
  • Active-Passive: Standby region for DR only
  • Follow-the-sun: Primary shifts based on time of day
11

What is Infrastructure as Code? Best practices at scale?

IaC manages infrastructure through code instead of manual processes.

Best practices at scale:

  • Modularization: Reusable modules for common patterns
  • State management: Remote state with locking (S3 + DynamoDB)
  • Environment separation: Workspaces or separate state files
  • Testing: terraform validate, tflint, terratest
  • GitOps: PR-based changes with automated plan/apply
  • Drift detection: Regular plan to detect manual changes
  • Secrets handling: External secrets management (Vault)
12

What is Virtualization? Types of hypervisors?

Virtualization creates virtual versions of computing resources (servers, storage, networks).

Type 1 (Bare-metal):

  • Runs directly on hardware
  • Better performance, more secure
  • Examples: VMware ESXi, Microsoft Hyper-V, KVM

Type 2 (Hosted):

  • Runs on top of host OS
  • Easier to set up, good for development
  • Examples: VirtualBox, VMware Workstation
13

Do we still need VMs in the age of containers?

Yes, VMs are still relevant:

  • Security isolation: Stronger boundaries than containers
  • Legacy applications: Apps that can't be containerized
  • Different OS: Running Windows workloads on Linux hosts
  • Compliance: Certain regulations require VM-level isolation
  • Kubernetes nodes: Containers run inside VMs

Emerging: Kata Containers, Firecracker - micro-VMs combining VM security with container speed.

14

How do you handle database scaling?

Read scaling:

  • Read replicas
  • Caching layer (Redis, Memcached)
  • Connection pooling (PgBouncer)

Write scaling:

  • Vertical scaling: Bigger instance
  • Sharding: Partition data across databases
  • CQRS: Separate read/write databases

Managed solutions: Aurora, Spanner, CockroachDB for automatic scaling.
