System Design Interview Q&A

Master system design questions covering distributed systems, microservices, caching, message queues, and architectural patterns.

System Design Questions

1

What is a "single point of failure"? How do you eliminate it?

A single point of failure (SPOF) is any component whose failure brings down the entire system.

Elimination strategies:

  • Redundancy at every layer (multiple instances, multi-AZ, multi-region)
  • Load balancers with health checks
  • Database replication (primary-replica, multi-master)
  • Stateless application design
  • Circuit breakers and graceful degradation
2

Explain CAP Theorem. How does it affect system design?

CAP Theorem states that a distributed system can only guarantee two of three properties:

  • Consistency: Every read receives the most recent write
  • Availability: Every request receives a response
  • Partition Tolerance: System operates despite network failures

In practice: Since network partitions are inevitable, you choose between CP (consistent but may be unavailable) or AP (available but may return stale data).

Examples: MongoDB (CP), Cassandra (AP), Traditional RDBMS (CA - no partition tolerance)

3

What is a CDN? Explain Multi-CDN architecture.

CDN (Content Delivery Network) is a geographically distributed network of servers that cache content close to end users.

Multi-CDN Benefits:

  • Better global coverage and performance
  • Increased redundancy and availability
  • Cost optimization through traffic steering
  • Protection against vendor outages

Implementation: Use DNS-based load balancing or a CDN orchestrator to route traffic to the optimal CDN based on performance, cost, or availability.

4

Explain "3-Tier Architecture" with pros and cons.

Three tiers:

  • Presentation Tier: Web servers, UI (Nginx, React)
  • Application Tier: Business logic (API servers, microservices)
  • Data Tier: Databases, storage (PostgreSQL, Redis)

Pros: Separation of concerns, independent scaling, better security isolation, easier maintenance.

Cons: Added complexity, potential latency between tiers, more infrastructure to manage.

5

Mono-repo vs. Multi-repo: When to use each?

Mono-repo (single repository for all code):

  • Pros: Atomic changes across services, unified tooling, easier code sharing
  • Cons: Requires specialized tooling (Bazel), can become slow, complex permissions
  • Used by: Google, Facebook, Microsoft

Multi-repo (separate repo per service):

  • Pros: Clear ownership, independent versioning, simpler CI/CD per repo
  • Cons: Dependency management challenges, harder cross-service changes
  • Better for: Distributed teams, microservices with clear boundaries
6

What are the drawbacks of monolithic architecture?

  • Scaling limitations: Must scale entire application even if only one component needs it
  • Deployment risk: Single change requires full deployment
  • Technology lock-in: Entire app uses same tech stack
  • Team bottlenecks: Multiple teams work on same codebase
  • Reliability: Bug in one module can crash entire application
  • Slow builds: As codebase grows, build times increase
7

What is a Service Mesh? When would you use it?

A service mesh is a dedicated infrastructure layer for handling service-to-service communication with features like:

  • Traffic management (routing, load balancing)
  • Security (mTLS, authentication)
  • Observability (metrics, tracing, logging)
  • Resiliency (retries, circuit breaking, timeouts)

Popular options: Istio, Linkerd, Consul Connect

Use when: You have many microservices and need standardized communication patterns, security, and observability without modifying application code.

8

Explain "Loose Coupling" and why it matters.

Loose coupling means components have minimal dependencies on each other and communicate through well-defined interfaces.

Techniques:

  • Event-driven architecture (pub/sub)
  • API contracts and versioning
  • Message queues between services
  • Service discovery instead of hardcoded endpoints

Benefits: Independent deployments, easier testing, better fault isolation, team autonomy.

9

What is a message queue? When is it used?

A message queue is an asynchronous communication mechanism between services.

Use cases:

  • Decoupling producers and consumers
  • Handling traffic spikes (buffering)
  • Background job processing
  • Event-driven architectures
  • Cross-service communication

Examples: RabbitMQ, Apache Kafka, AWS SQS, Redis Streams

10

What are cache replacement policies? Explain LRU.

Cache replacement policies determine which items to evict when cache is full:

  • LRU (Least Recently Used): Evicts items not accessed for longest time. Best for temporal locality.
  • LFU (Least Frequently Used): Evicts items with fewest accesses. Good for popularity-based access.
  • FIFO (First In, First Out): Evicts oldest items. Simple but not optimal.
  • TTL (Time To Live): Items expire after set duration.

LRU implementation: Typically uses a hash map + doubly linked list for O(1) get/put operations.

11

Design a URL shortener. What are the key considerations?

Key components:

  • URL encoding: Base62 encoding (a-z, A-Z, 0-9) for short codes
  • ID generation: Counter-based, UUID, or distributed ID generator (Snowflake)
  • Storage: Key-value store (Redis) for hot data, database for persistence
  • Caching: Cache popular URLs at edge/CDN level

Considerations:

  • Custom aliases vs. auto-generated
  • Analytics and click tracking
  • Rate limiting and abuse prevention
  • Expiration policies
  • Scale: Reads >> Writes (cache heavily)
12

Explain the Sidecar Pattern. Give examples.

The sidecar pattern deploys a helper container alongside the main application container to provide supporting features.

Use cases:

  • Logging: Fluentd sidecar collecting logs
  • Proxy: Envoy sidecar for service mesh (Istio)
  • Monitoring: Prometheus exporter sidecar
  • Security: Vault agent for secrets injection
  • Sync: Git-sync sidecar for config updates

Benefits: Separation of concerns, independent lifecycle, language-agnostic features.

13

What is "Consistent Hashing"? Where is it used?

Consistent hashing is a technique for distributing data across nodes where adding/removing nodes only requires remapping a small portion of keys.

How it works: Nodes and keys are placed on a virtual ring. Each key is assigned to the first node clockwise from its position.

Used in:

  • Distributed caches (Memcached, Redis Cluster)
  • Load balancers (sticky sessions)
  • Distributed databases (Cassandra, DynamoDB)
  • CDNs for content routing

Improvement: Virtual nodes (vnodes) for better load distribution.

14

What is "Branch by Abstraction" for migrations?

Branch by Abstraction is a technique for making large-scale changes gradually without long-lived feature branches.

Steps:

  1. Create an abstraction layer over the code to be replaced
  2. Migrate all clients to use the abstraction
  3. Build new implementation behind the abstraction
  4. Gradually switch traffic to new implementation
  5. Remove old implementation and abstraction

Benefits: Continuous integration, easy rollback, parallel development.

15

What is "Shared-Nothing" architecture?

Shared-Nothing architecture means each node is independent and self-sufficient with its own CPU, memory, and storage.

Characteristics:

  • No shared state between nodes
  • Communication only through network
  • Horizontal scalability by adding nodes
  • No single point of failure

Examples: Cassandra, ScyllaDB, Citus, most microservices architectures

Contrast: Shared-disk (Oracle RAC), Shared-memory architectures

16

How would you design a video streaming platform?

Key components:

  • Ingestion: Video upload, transcoding pipeline (different resolutions/codecs)
  • Storage: Object storage (S3) for video files, metadata in database
  • CDN: Global edge caching for low latency playback
  • Adaptive bitrate: HLS/DASH protocols for quality adjustment
  • APIs: Catalog service, user service, recommendation engine

Scaling considerations:

  • Read-heavy: Cache aggressively at CDN and application layer
  • Separate upload and streaming paths
  • Async processing for transcoding (queue-based)
  • Regional deployments for latency
