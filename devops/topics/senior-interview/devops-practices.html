DevOps Practices Questions

1

What is DevOps? How does it differ from traditional IT?

DevOps is a culture and set of practices that unifies software development (Dev) and IT operations (Ops).

Key differences from traditional IT:

  • Culture: Collaboration vs. silos; shared responsibility
  • Speed: Continuous delivery vs. quarterly releases
  • Automation: Everything as code vs. manual processes
  • Feedback: Fast loops vs. long cycles
  • Risk: Small, frequent changes vs. big-bang releases
2

What is SRE (Site Reliability Engineering)? How does it relate to DevOps?

SRE is Google's approach to operations, treating operations as a software engineering problem.

Key concepts:

  • Error budgets: Acceptable level of unreliability (100% - SLO)
  • SLIs/SLOs/SLAs: Measurable reliability targets
  • Toil reduction: Automate repetitive operational work
  • Blameless postmortems: Learn from failures

Relationship to DevOps: SRE is one implementation of DevOps. "DevOps is what you do, SRE is how Google does it."

3

What is GitOps? Benefits and challenges?

GitOps uses Git as the single source of truth for declarative infrastructure and applications.

Principles:

  • Declarative descriptions in Git
  • Desired state versioned and immutable
  • Automated agents pull changes (ArgoCD, Flux)
  • Continuous reconciliation

Benefits: Audit trail, easy rollback, consistent environments, PR-based workflow.

Challenges: Secrets management, handling drift, learning curve.

4

Explain Blue-Green vs. Canary vs. Rolling deployments.

Blue-Green:

  • Two identical environments; switch traffic instantly
  • Pros: Instant rollback, no downtime
  • Cons: 2x infrastructure cost

Canary:

  • Gradually shift traffic (1% → 10% → 50% → 100%)
  • Pros: Lower risk, real-world testing
  • Cons: Longer deployment, complex traffic management

Rolling:

  • Replace instances one by one
  • Pros: No extra infrastructure
  • Cons: Mixed versions during deployment
5

What is Platform Engineering? How does it differ from DevOps?

Platform Engineering builds internal developer platforms (IDPs) to reduce cognitive load on developers.

Key aspects:

  • Self-service infrastructure
  • Golden paths and templates
  • Developer experience focus
  • Internal product mindset

Difference from DevOps: DevOps focuses on culture and practices; Platform Engineering creates products that enable DevOps at scale.

Tools: Backstage, Port, Humanitec

6

What is Continuous Integration? Best practices?

Continuous Integration (CI) is the practice of frequently merging code changes into a shared repository with automated testing.

Best practices:

  • Commit frequently (at least daily)
  • Automated builds on every commit
  • Fast feedback (< 10 minutes for basic tests)
  • Fix broken builds immediately
  • Keep main branch always deployable
  • Test in production-like environments
  • Make build status visible to team
7

Continuous Delivery vs. Continuous Deployment?

Continuous Delivery:

  • Every change is releasable to production
  • Manual approval gate before production
  • Business decides when to release

Continuous Deployment:

  • Every change that passes tests goes to production automatically
  • No manual intervention
  • Requires mature testing and monitoring

Key: CD (Delivery) is about capability; CD (Deployment) is about actually doing it.

8

What types of testing should be in a CI/CD pipeline?

Testing pyramid (bottom to top):

  • Unit tests: Fast, isolated, high coverage
  • Integration tests: Component interactions
  • API/Contract tests: Service boundaries
  • E2E tests: Full user journeys (fewer, slower)

Additional tests:

  • Static analysis: Linting, code quality
  • Security scanning: SAST, dependency checks
  • Performance tests: Load testing on critical paths
  • Smoke tests: Quick post-deployment verification
9

What is "Shift Left"? How do you implement it?

Shift Left means moving testing, security, and quality checks earlier in the development lifecycle.

Implementation:

  • Pre-commit hooks: Linting, formatting, basic tests
  • IDE integration: Security scanners, code analysis
  • PR checks: Required tests before merge
  • Early security: SAST in CI, dependency scanning
  • Design reviews: Threat modeling, architecture review

Benefits: Cheaper to fix (10x cost increase per stage), faster feedback, better quality.

10

What are Feature Flags? When to use them?

Feature flags (toggles) are runtime switches to enable/disable features without deploying.

Use cases:

  • Release flags: Deploy code but enable later
  • Experiment flags: A/B testing
  • Operational flags: Kill switch for problematic features
  • Permission flags: Enable for specific users/groups

Best practices: Short-lived flags, track flag debt, remove after rollout, audit flag changes.

Tools: LaunchDarkly, Split, Flagsmith, custom implementations.

11

What is Trunk-Based Development?

Trunk-Based Development is a branching model where developers commit to a single branch (trunk/main).

Characteristics:

  • Short-lived feature branches (< 1 day)
  • No long-running branches
  • Feature flags for incomplete features
  • Requires strong CI and testing

Benefits: Reduced merge conflicts, faster integration, always-releasable main.

Contrast: GitFlow has multiple long-lived branches (develop, release, hotfix).

12

How do you handle database migrations in CI/CD?

Best practices:

  • Versioned migrations: Sequential, idempotent scripts
  • Backward compatible: New code works with old schema
  • Expand/Contract pattern: Add new, migrate data, remove old
  • Blue-green for DB: Shadow database for testing

Tools: Flyway, Liquibase, Rails migrations, Alembic

Key rule: Deploy migrations separately from application changes; never make breaking schema changes.

13

What is a blameless postmortem?

Blameless postmortem is a retrospective after an incident focusing on systemic improvements, not individual blame.

Structure:

  • Timeline: What happened and when
  • Impact: Users affected, duration
  • Root cause: 5 Whys analysis
  • Contributing factors: What made it worse
  • Action items: Preventive measures with owners

Philosophy: "How did the system allow this to happen?" not "Who caused this?"

14

What is Chaos Engineering?

Chaos Engineering is the discipline of experimenting on systems to build confidence in their resilience.

Principles:

  • Define steady state (normal behavior)
  • Hypothesize steady state continues during experiment
  • Introduce real-world failures
  • Minimize blast radius

Examples: Kill random pods, inject latency, simulate AZ failure, exhaust resources.

Tools: Chaos Monkey, Gremlin, Litmus, Chaos Mesh.

15

How do you measure DevOps success?

DORA Metrics (from "Accelerate" book):

  • Deployment Frequency: How often you deploy to production
  • Lead Time for Changes: Commit to production time
  • Mean Time to Recovery (MTTR): Time to restore service
  • Change Failure Rate: % of deployments causing failures

Additional metrics:

  • Availability/SLO attainment
  • Developer satisfaction
  • Time to onboard new developers
  • Security vulnerability remediation time
