Modules & Packages

A module is a .py file. A package is a directory with __init__.py (or namespace package). Imports follow sys.path resolution.

💡

Modules = single tools. Packages = toolboxes containing related tools.

Key Concepts

Module

Single .py file

Package

Directory with __init__.py

Absolute vs Relative

import myapp.utils vs from . import utils

__init__.py

Marks dir as package; can run init code

Namespace Package

Package split across directories (PEP 420)

Common Patterns

from datetime import datetime Import specific name
import requests as r Aliased import
if __name__ == '__main__': main() Run-as-script idiom
python -m mypackage Run package's __main__.py
python -c 'import sys; print(sys.path)' Show import paths

Interview Questions

1

Module vs package?

Module = single .py file. Package = directory with __init__.py containing modules. Packages can be nested.
2

__init__.py purpose?

Marks directory as a regular package, runs initialization code, can re-export names for convenience.
3

Absolute vs relative imports?

Absolute: 'from myapp.utils import helper'. Relative: 'from .utils import helper' (within same package).
4

Why 'if __name__ == "__main__":'?

Runs code only when script is executed directly, not when imported. Allows files to be both modules and scripts.
5

How does Python find imports?

Searches sys.path: current dir, PYTHONPATH, site-packages. First match wins.

Real-World Scenarios

Circular import error

🔍 Approach
  1. Refactor: extract shared code into a 3rd module
  2. Move imports into functions (lazy)
  3. Avoid module-level side effects
  4. Use forward references with type hints

ModuleNotFoundError in production

🔍 Approach
  1. Check sys.path includes app dir
  2. Verify package installed in venv
  3. Use pip freeze to compare envs
  4. Use 'python -m' for executing modules

Quick Tips

Always include __init__.py for clarity

Prefer absolute imports

Use 'if __name__ == "__main__":' guard

Use 'python -m' for portable execution
