What is Jenkins?

Jenkins is an open-source automation server written in Java. It helps automate building, testing, and deploying software, enabling continuous integration and continuous delivery (CI/CD).

🏭

Factory Analogy

Think of Jenkins as a factory manager. It receives raw materials (code), orchestrates assembly lines (pipelines), quality checks (tests), and ships finished products (deployments) - all automatically!

Architecture Components:

Master (Controller)

Schedules builds, dispatches to agents, monitors agents, records results, serves UI

Agent (Node)

Executes build jobs. Can be permanent or dynamic (Docker, Kubernetes, Cloud)

Executor

A slot for running a job on an agent. Each agent can have multiple executors

Plugins

Extend Jenkins functionality: Git, Docker, Kubernetes, Slack, SonarQube, and 1800+ more

Key Concepts

Job / Project

A runnable task configured in Jenkins - Freestyle, Pipeline, Multi-branch, etc.

Build

A single execution of a job. Has a number, status, logs, and artifacts

Pipeline

Suite of plugins for implementing CI/CD as code (Jenkinsfile)

Stage

A logical grouping of steps (Build, Test, Deploy) shown in Blue Ocean

Step

A single task: checkout code, run shell, build Docker image

Workspace

Directory on agent where build executes. Contains source, artifacts

Trigger

Event that starts a build: SCM poll, webhook, cron, upstream job

Artifact

Files produced by a build: JARs, Docker images, test reports

Pipeline Syntax

Declarative Pipeline (Recommended)

Jenkinsfile 
pipeline {
    agent any
    
    environment {
        DOCKER_REGISTRY = 'registry.example.com'
        APP_NAME = 'my-app'
    }
    
    options {
        timeout(time: 30, unit: 'MINUTES')
        disableConcurrentBuilds()
        buildDiscarder(logRotator(numToKeepStr: '10'))
    }
    
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        
        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }
        
        stage('Test') {
            parallel {
                stage('Unit Tests') {
                    steps {
                        sh 'mvn test'
                    }
                }
                stage('Integration Tests') {
                    steps {
                        sh 'mvn verify -Pintegration'
                    }
                }
            }
        }
        
        stage('SonarQube Analysis') {
            steps {
                withSonarQubeEnv('SonarQube') {
                    sh 'mvn sonar:sonar'
                }
            }
        }
        
        stage('Build Docker Image') {
            steps {
                script {
                    docker.build("${DOCKER_REGISTRY}/${APP_NAME}:${BUILD_NUMBER}")
                }
            }
        }
        
        stage('Deploy to Staging') {
            when {
                branch 'develop'
            }
            steps {
                sh 'kubectl apply -f k8s/staging/'
            }
        }
        
        stage('Deploy to Production') {
            when {
                branch 'main'
            }
            input {
                message "Deploy to production?"
                ok "Yes, deploy!"
            }
            steps {
                sh 'kubectl apply -f k8s/production/'
            }
        }
    }
    
    post {
        always {
            junit '**/target/surefire-reports/*.xml'
            archiveArtifacts artifacts: '**/target/*.jar', fingerprint: true
        }
        success {
            slackSend channel: '#deployments', color: 'good',
                      message: "Build ${BUILD_NUMBER} succeeded"
        }
        failure {
            slackSend channel: '#deployments', color: 'danger',
                      message: "Build ${BUILD_NUMBER} failed"
        }
        cleanup {
            cleanWs()
        }
    }
}

Scripted Pipeline

Jenkinsfile (Scripted) 
node('linux') {
    try {
        stage('Checkout') {
            checkout scm
        }
        
        stage('Build') {
            sh 'mvn clean package'
        }
        
        stage('Test') {
            sh 'mvn test'
        }
        
        stage('Deploy') {
            if (env.BRANCH_NAME == 'main') {
                sh 'kubectl apply -f deployment.yaml'
            }
        }
    } catch (Exception e) {
        currentBuild.result = 'FAILURE'
        throw e
    } finally {
        junit '**/target/surefire-reports/*.xml'
        cleanWs()
    }
}

Declarative - Structured, easier to read, better error handling, recommended for most cases

Scripted - Full Groovy power, more flexible, useful for complex logic

Shared Libraries

Shared Libraries allow you to define reusable pipeline code in a separate repository, promoting DRY principles and standardization across teams.

Library Structure:

Directory Structure 
shared-library/
├── vars/
│   ├── buildJavaApp.groovy    # Global variables
│   └── deployToK8s.groovy
├── src/
│   └── com/company/
│       └── Utils.groovy       # Classes
└── resources/
    └── templates/             # Static files

Custom Step (vars/buildJavaApp.groovy):

vars/buildJavaApp.groovy 
def call(Map config = [:]) {
    pipeline {
        agent any
        stages {
            stage('Build') {
                steps {
                    sh "mvn clean package -DskipTests=${config.skipTests ?: false}"
                }
            }
            stage('Test') {
                when { expression { !config.skipTests } }
                steps {
                    sh 'mvn test'
                }
            }
        }
    }
}

Usage in Jenkinsfile:

Jenkinsfile 
@Library('my-shared-library') _

buildJavaApp(skipTests: false)

Security Best Practices

Use Credentials Plugin - Never hardcode secrets in Jenkinsfiles or job configs

Enable CSRF Protection - Manage Jenkins → Configure Global Security

Role-Based Access Control - Use Role Strategy Plugin for fine-grained permissions

Agent-to-Master Security - Enable in security settings to restrict agent access

Audit Trail - Enable audit logging plugin to track changes

Script Approval - Review and approve Groovy scripts in sandbox

Network Isolation - Run Jenkins in private subnet, expose via reverse proxy

Regular Updates - Keep Jenkins and plugins updated for security patches

Real-World Scenarios

Multi-Branch Pipeline for Microservices

Problem:

Team has 20+ microservices, each with multiple feature branches. Need automated CI for all branches.

Solution:
  1. Create Jenkinsfile in each repo root
  2. Configure Multi-branch Pipeline job in Jenkins
  3. Set branch discovery: all branches
  4. Configure webhook in GitHub/GitLab
  5. Jenkins auto-discovers branches and creates jobs
Result:

Every push triggers CI, branches are tested in isolation, PRs show build status.

Dynamic Kubernetes Agents

Problem:

Static agents are expensive and don't scale with build demand.

Solution:
pipeline {
    agent {
        kubernetes {
            yaml '''
            apiVersion: v1
            kind: Pod
            spec:
              containers:
              - name: maven
                image: maven:3.8-openjdk-11
                command: ['sleep', 'infinity']
              - name: docker
                image: docker:dind
                securityContext:
                  privileged: true
            '''
        }
    }
    stages {
        stage('Build') {
            steps {
                container('maven') {
                    sh 'mvn package'
                }
            }
        }
    }
}
Result:

Agents spin up on demand, costs reduced by 60%, infinite parallelism.

Interview Questions

1

What is the difference between Declarative and Scripted Pipeline?

  • Declarative: Structured syntax with predefined sections (pipeline, stages, steps). Easier to read, built-in validation, better error messages. Recommended for most use cases.
  • Scripted: Full Groovy DSL, more flexible, uses node/stage blocks. Better for complex logic but harder to maintain.

Pro Tip: Start with Declarative, use script {} blocks when you need Groovy flexibility.

2

How do you secure sensitive credentials in Jenkins?

  • Use Credentials Plugin to store secrets encrypted
  • Access via withCredentials() block - secrets are masked in logs
  • Use Vault or AWS Secrets Manager for enterprise
  • Never echo credentials: set +x in shell scripts
  • Use credential types: Username/Password, Secret Text, SSH Key, Certificate
withCredentials([string(credentialsId: 'api-key', variable: 'TOKEN')]) { sh 'curl -H "Authorization: Bearer $TOKEN" https://api.example.com' }
3

Explain Jenkins Master-Agent architecture.

  • Master (Controller): Schedules builds, monitors agents, serves UI, stores config. Should NOT run builds in production.
  • Agent (Node): Executes builds. Can be permanent (VM) or ephemeral (Docker/K8s pod).
  • Communication: Agent connects to master via JNLP (port 50000) or SSH.
  • Executors: Number of concurrent builds per agent.

Best Practice: Set master executors to 0. All builds should run on agents for security and scalability.

4

What are Shared Libraries and why use them?

  • Definition: Reusable Groovy code stored in Git, loaded into pipelines.
  • Benefits: DRY principle, standardization, version control, easier maintenance.
  • Structure: vars/ for global functions, src/ for classes, resources/ for static files.
  • Usage: @Library('my-lib') _ at top of Jenkinsfile.
5

How do you implement GitOps with Jenkins?

  1. Store Jenkinsfile in application repo (CI config as code)
  2. Use Multi-branch Pipeline to auto-discover branches
  3. Configure webhooks for push-triggered builds
  4. Store K8s manifests in Git, apply via pipeline
  5. Use git tag for release versioning
  6. Infrastructure changes via PR + pipeline approval
6

How do you handle pipeline failures and retries?

  • post block: Define actions for success, failure, always, cleanup
  • retry: retry(3) { sh 'flaky-command' }
  • timeout: timeout(time: 10, unit: 'MINUTES') { ... }
  • catchError: Continue pipeline even if stage fails
  • input: Manual intervention for critical failures
stage('Deploy') { steps { retry(3) { timeout(time: 5, unit: 'MINUTES') { sh 'kubectl apply -f deployment.yaml' } } } }
7

What is Blue Ocean and when would you use it?

  • Blue Ocean: Modern, visual UI for Jenkins pipelines
  • Features: Pipeline visualization, visual editor, GitHub/Bitbucket integration, personalized dashboard
  • Use when: Creating pipelines visually, debugging complex multi-branch pipelines, presenting to stakeholders
  • Limitation: Some advanced features only in classic UI
8

How do you optimize Jenkins performance?

  • Distributed builds: Use agents, never build on master
  • Build discarder: Limit build history retention
  • Workspace cleanup: cleanWs() in post section
  • Parallel stages: Run independent stages concurrently
  • Artifact management: Use external artifact storage (Nexus, S3)
  • Plugin hygiene: Remove unused plugins, keep updated
  • JVM tuning: Increase heap size for large instances
9

Explain the difference between Freestyle and Pipeline jobs.

  • Freestyle: UI-configured, single build task, limited flexibility, hard to version control
  • Pipeline: Code-defined (Jenkinsfile), complex workflows, version controlled, reusable

Recommendation: Always use Pipeline for new projects. Freestyle is legacy.

10

How do you implement approval gates in Jenkins?

stage('Deploy to Prod') { input { message "Deploy to production?" ok "Yes, deploy!" submitter "admin,release-team" parameters { choice(name: 'ENV', choices: ['us-east', 'eu-west'], description: 'Target region') } } steps { sh "kubectl apply -f k8s/${ENV}/" } }
  • input: Pauses pipeline for manual approval
  • submitter: Restrict who can approve
  • timeout: Auto-abort if not approved in time
11

What is Jenkins Configuration as Code (JCasC)?

  • JCasC: Plugin to configure Jenkins via YAML files
  • Benefits: Reproducible setup, version controlled, disaster recovery, no manual UI config
  • Covers: Security realm, authorization, credentials, tools, system settings, plugins
jenkins: systemMessage: "Jenkins configured by JCasC" securityRealm: ldap: server: "ldap.company.com" authorizationStrategy: roleBased: roles: global: - name: "admin" permissions: - "Overall/Administer"
12

How do you troubleshoot a failed Jenkins build?

  1. Check console output: Full build log with error details
  2. Pipeline Steps: View individual step durations and failures
  3. Replay: Re-run with modified Jenkinsfile for debugging
  4. Workspace: Inspect files on agent
  5. Blue Ocean: Visual pipeline view shows failed stage
  6. System logs: Manage Jenkins → System Log
  7. SSH to agent: Debug environment issues directly
13

What are Jenkins webhooks and how do they work?

  • Purpose: Trigger builds immediately on code push (vs. polling)
  • Setup: Configure webhook URL in GitHub/GitLab pointing to Jenkins
  • URL format: https://jenkins.example.com/github-webhook/
  • Authentication: Use personal access token or dedicated user
  • Events: Push, PR opened, tag created, etc.

Tip: Webhooks are more efficient than SCM polling. Builds start instantly and reduce Jenkins load.

14

How do you run parallel stages in Jenkins Pipeline?

stage('Test') { parallel { stage('Unit Tests') { agent { label 'linux' } steps { sh 'npm run test:unit' } } stage('Integration Tests') { agent { label 'linux' } steps { sh 'npm run test:integration' } } stage('E2E Tests') { agent { label 'chrome' } steps { sh 'npm run test:e2e' } } } failFast true // Fail immediately if any parallel stage fails }
  • failFast: Stop all parallel stages if one fails
  • Different agents: Each parallel stage can use different agents
15

What is the Groovy Sandbox in Jenkins?

  • Purpose: Security layer that restricts what Groovy code can do
  • How it works: Pipeline scripts run in sandbox by default, blocking dangerous operations
  • Script Approval: Admin must approve scripts that need elevated permissions
  • Location: Manage Jenkins → In-process Script Approval

Security: Be cautious approving scripts. Malicious code can compromise Jenkins server.

16

How do you back up Jenkins?

  • JENKINS_HOME: Backup entire directory (/var/lib/jenkins)
  • Critical folders: jobs/, config.xml, users/, secrets/, plugins/
  • Exclude: workspace/, builds/ (large, reproducible)
  • Plugins: ThinBackup, Periodic Backup for automated backups
  • JCasC: Export config to YAML, store in Git
# Backup script tar -czvf jenkins-backup-$(date +%F).tar.gz \ --exclude='workspace' --exclude='builds' \ /var/lib/jenkins/
17

How do you integrate Jenkins with SonarQube?

  1. Install SonarQube Scanner plugin
  2. Configure SonarQube server in Manage Jenkins → Configure System
  3. Add SonarQube token as credentials
  4. Use in pipeline:
stage('SonarQube Analysis') { steps { withSonarQubeEnv('SonarQube') { sh 'mvn sonar:sonar' } } } stage('Quality Gate') { steps { timeout(time: 5, unit: 'MINUTES') { waitForQualityGate abortPipeline: true } } }
18

What is the difference between agent any, agent none, and agent label?

  • agent any: Run on any available agent
  • agent none: No global agent; each stage must specify its own
  • agent { label 'linux' }: Run only on agents with matching label
  • agent { docker 'maven:3.8' }: Run in Docker container
  • agent { kubernetes { ... } }: Dynamic K8s pod
19

How do you implement blue-green deployment with Jenkins?

stage('Deploy Green') { steps { sh 'kubectl apply -f green-deployment.yaml' sh 'kubectl wait --for=condition=available deployment/app-green' } } stage('Smoke Test Green') { steps { sh 'curl -f http://green.internal/health' } } stage('Switch Traffic') { input { message "Switch traffic to green?" } steps { sh 'kubectl patch service app -p \'{"spec":{"selector":{"version":"green"}}}\'' } } stage('Cleanup Blue') { steps { sh 'kubectl delete deployment app-blue' } }
20

How do you handle environment-specific configurations?

pipeline { agent any parameters { choice(name: 'ENV', choices: ['dev', 'staging', 'prod'], description: 'Target environment') } environment { CONFIG_FILE = "config-${params.ENV}.yaml" } stages { stage('Deploy') { steps { withCredentials([file(credentialsId: "${params.ENV}-kubeconfig", variable: 'KUBECONFIG')]) { sh "kubectl apply -f k8s/${params.ENV}/" } } } } }
  • Use parameters for environment selection
  • Store environment-specific credentials separately
  • Keep config in separate folders or files per environment
