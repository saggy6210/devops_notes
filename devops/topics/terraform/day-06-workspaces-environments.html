Workspaces & Environments

🌍 Workspaces & Environment Management

Master Terraform workspaces and learn different strategies for managing multiple environments like dev, staging, and production.

Terraform Workspaces

Workspaces allow you to manage multiple instances of a single configuration with separate state files.

Workspace Commands 
# List workspaces (* indicates current)
terraform workspace list
# * default
#   dev
#   staging

# Create new workspace
terraform workspace new dev

# Switch workspace
terraform workspace select dev

# Show current workspace
terraform workspace show

# Delete workspace
terraform workspace delete dev

Using Workspace in Configuration

Workspace-Aware Config 
locals {
  environment = terraform.workspace

  instance_type = {
    default = "t3.micro"
    dev     = "t3.micro"
    staging = "t3.small"
    prod    = "t3.large"
  }

  instance_count = {
    default = 1
    dev     = 1
    staging = 2
    prod    = 3
  }
}

resource "aws_instance" "web" {
  count         = local.instance_count[terraform.workspace]
  instance_type = local.instance_type[terraform.workspace]
  ami           = var.ami_id

  tags = {
    Name        = "web-${terraform.workspace}-${count.index + 1}"
    Environment = terraform.workspace
  }
}

Workspace Limitations

Workspaces share the same backend, providers, and configuration. They're best for minor variations, not completely different infrastructures.

Directory-Based Environments

Separate directories for each environment provide complete isolation and different configurations.

Directory Structure 
infrastructure/
├── modules/                    # Shared modules
│   ├── vpc/
│   ├── ec2/
│   └── rds/
├── environments/
│   ├── dev/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── terraform.tfvars
│   │   └── backend.tf
│   ├── staging/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── terraform.tfvars
│   │   └── backend.tf
│   └── prod/
│       ├── main.tf
│       ├── variables.tf
│       ├── terraform.tfvars
│       └── backend.tf
└── global/                     # Shared resources
    └── iam/
environments/dev/main.tf 
module "vpc" {
  source = "../../modules/vpc"

  environment = "dev"
  vpc_cidr    = var.vpc_cidr
}

module "web" {
  source = "../../modules/ec2"

  environment   = "dev"
  instance_type = "t3.micro"
  instance_count = 1
  subnet_ids    = module.vpc.public_subnet_ids
}
environments/dev/backend.tf 
terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state"
    key            = "dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}

Variable Files per Environment

Use separate .tfvars files for each environment with a single configuration.

Structure 
project/
├── main.tf
├── variables.tf
├── outputs.tf
└── environments/
    ├── dev.tfvars
    ├── staging.tfvars
    └── prod.tfvars
environments/dev.tfvars 
environment    = "dev"
instance_type  = "t3.micro"
instance_count = 1
vpc_cidr       = "10.1.0.0/16"
enable_monitoring = false
environments/prod.tfvars 
environment    = "prod"
instance_type  = "t3.large"
instance_count = 3
vpc_cidr       = "10.0.0.0/16"
enable_monitoring = true
Usage 
# Apply for dev
terraform plan -var-file="environments/dev.tfvars"
terraform apply -var-file="environments/dev.tfvars"

# Apply for prod
terraform plan -var-file="environments/prod.tfvars"
terraform apply -var-file="environments/prod.tfvars"

Strategy Comparison

Approach Pros Cons Best For
Workspaces Simple, single config, shared code Same backend, limited isolation Similar environments
Directories Full isolation, different configs Code duplication, harder maintenance Different architectures
Var Files Simple, single codebase Must manage state carefully Configuration differences only

Terraform Cloud Workspaces

Terraform Cloud provides enhanced workspaces with remote execution, collaboration, and policy enforcement.

Terraform Cloud Backend 
terraform {
  cloud {
    organization = "my-org"

    workspaces {
      name = "my-app-prod"
    }
  }
}

# Or use workspace tags for multiple workspaces
terraform {
  cloud {
    organization = "my-org"

    workspaces {
      tags = ["app:myapp", "env:prod"]
    }
  }
}

Interview Q&A

Q1: What are Terraform workspaces?

Workspaces allow multiple state instances for the same configuration:

  • Each workspace has its own state file
  • Default workspace always exists
  • Access current workspace via terraform.workspace
  • Useful for similar environments (dev/staging/prod)
Q2: When should you NOT use workspaces?
  • Environments need different backends
  • Significantly different configurations
  • Different providers per environment
  • Different access controls needed
  • Production isolation requirements
Q3: How do you manage multiple environments?

Common strategies:

  • Workspaces: Single config, multiple states
  • Directories: Separate folders per environment
  • Var files: Different .tfvars per environment
  • Terraform Cloud: Managed workspaces with RBAC
Q4: What is Terraform Cloud?

Terraform Cloud is HashiCorp's managed service:

  • Remote state management with encryption
  • Remote plan/apply execution
  • Team collaboration features
  • Policy enforcement (Sentinel)
  • VCS integration for GitOps
  • Private module registry
Q5: How do you promote changes between environments?
  • Use modules with versioning
  • Test in dev → staging → prod
  • Git branches per environment
  • CI/CD pipelines with approvals
  • Use same module versions across environments
