Providers & Resources

🔌 Providers & Resources

Understand how Terraform interacts with cloud platforms through providers and how to define infrastructure resources.

Understanding Providers

Providers are plugins that Terraform uses to interact with cloud platforms, SaaS providers, and other APIs. Each provider adds a set of resource types and data sources.

AWS Provider

EC2, S3, RDS, Lambda, VPC, and 500+ AWS services.

Azure Provider

VMs, Storage, AKS, Functions, and Azure services.

GCP Provider

Compute Engine, GKE, Cloud Storage, BigQuery.

Kubernetes Provider

Deploy and manage Kubernetes resources.

Provider Configuration

Basic Provider Block

providers.tf 
# AWS Provider
provider "aws" {
  region  = "us-east-1"
  profile = "default"
}

# Azure Provider
provider "azurerm" {
  features {}
  subscription_id = "your-subscription-id"
}

# GCP Provider
provider "google" {
  project = "my-project-id"
  region  = "us-central1"
}

Provider Version Constraints

versions.tf 
terraform {
  required_version = ">= 1.0.0"
  
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"    # >= 5.0, < 6.0
    }
    azurerm = {
      source  = "hashicorp/azurerm"
      version = ">= 3.0.0"  # Any version >= 3.0.0
    }
    random = {
      source  = "hashicorp/random"
      version = "3.5.1"     # Exact version
    }
  }
}

Version Constraint Syntax

Constraint Meaning
= 3.0.0 Exact version only
>= 3.0.0 Version 3.0.0 or greater
~> 3.0 Any 3.x version (>= 3.0, < 4.0)
~> 3.1.0 Any 3.1.x version (>= 3.1.0, < 3.2.0)
>= 3.0, < 4.0 Between 3.0 and 4.0

Multiple Provider Configurations

Use alias to create multiple configurations of the same provider for multi-region or multi-account deployments.

Multi-Region AWS 
# Default provider (us-east-1)
provider "aws" {
  region = "us-east-1"
}

# Alternative provider for EU
provider "aws" {
  alias  = "eu"
  region = "eu-west-1"
}

# Alternative provider for Asia
provider "aws" {
  alias  = "asia"
  region = "ap-southeast-1"
}

# Use default provider
resource "aws_s3_bucket" "us_bucket" {
  bucket = "my-us-bucket"
}

# Use EU provider
resource "aws_s3_bucket" "eu_bucket" {
  provider = aws.eu
  bucket   = "my-eu-bucket"
}

# Use Asia provider
resource "aws_s3_bucket" "asia_bucket" {
  provider = aws.asia
  bucket   = "my-asia-bucket"
}

Understanding Resources

Resources are the most important element in Terraform. Each resource block describes one or more infrastructure objects.

Resource Syntax 
# Syntax: resource "TYPE" "NAME" { ... }

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  
  tags = {
    Name        = "WebServer"
    Environment = "Production"
  }
}

Resource Address

Each resource has a unique address: <type>.<name>

  • aws_instance.web_server - The resource address
  • aws_instance.web_server.id - Instance ID attribute
  • aws_instance.web_server.public_ip - Public IP attribute

EC2 Instance

EC2 Instance with VPC

Data Sources

Data sources allow Terraform to fetch information from external sources or existing resources not managed by the current configuration.

Data Source Examples 
# Get latest Amazon Linux 2 AMI
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]
  
  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
  
  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

# Use the data source in a resource
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"
}

# Get existing VPC
data "aws_vpc" "existing" {
  id = "vpc-12345678"
}

# Get available AZs
data "aws_availability_zones" "available" {
  state = "available"
}

# Get current AWS account ID
data "aws_caller_identity" "current" {}

output "account_id" {
  value = data.aws_caller_identity.current.account_id
}

Resource vs Data Source

Resource Data Source
Creates/manages infrastructure Reads existing infrastructure
resource "aws_vpc" "main" data "aws_vpc" "existing"
Tracks in state file Only stores read values
Apply creates/updates Plan/apply only reads

Resource Dependencies

Implicit Dependencies

Terraform automatically detects dependencies when you reference one resource's attribute in another.

Implicit Dependency 
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

# Implicitly depends on aws_vpc.main
resource "aws_subnet" "public" {
  vpc_id     = aws_vpc.main.id  # References VPC
  cidr_block = "10.0.1.0/24"
}

Explicit Dependencies

Use depends_on when dependencies aren't captured by resource references.

Explicit Dependency 
resource "aws_iam_role" "lambda_role" {
  name = "lambda_role"
  # ... role configuration
}

resource "aws_iam_role_policy" "lambda_policy" {
  role = aws_iam_role.lambda_role.id
  # ... policy configuration
}

resource "aws_lambda_function" "app" {
  function_name = "my_function"
  role          = aws_iam_role.lambda_role.arn
  
  # Explicitly wait for policy to be attached
  depends_on = [aws_iam_role_policy.lambda_policy]
}

Interview Q&A

Q1: What is a Terraform provider?

A provider is a plugin that Terraform uses to interact with APIs of cloud platforms and services.

  • Each provider adds resource types and data sources
  • Configured in the provider block
  • Downloaded during terraform init
  • Examples: aws, azurerm, google, kubernetes
  • Providers are versioned and should be locked
Q2: How do you use multiple AWS regions in Terraform?

Use provider aliases to configure multiple regions:

  • Define default provider for primary region
  • Create aliased providers for additional regions
  • Use provider = aws.alias in resources
  • Each alias can have different credentials/configs
Q3: What is the difference between a resource and a data source?

Resource: Creates and manages infrastructure. Terraform tracks it in state and can update/delete it.

Data Source: Reads information from existing resources not managed by current configuration. Read-only, doesn't create anything.

  • Resource: resource "aws_vpc" "new" {}
  • Data: data "aws_vpc" "existing" {}
Q4: How does Terraform handle resource dependencies?

Terraform handles dependencies in two ways:

  • Implicit: Automatically detected when one resource references another's attributes
  • Explicit: Declared using depends_on meta-argument
  • Terraform builds a dependency graph to determine order
  • Independent resources are created/destroyed in parallel
Q5: What is the ~> version constraint in Terraform?

The ~> (pessimistic constraint) allows only the rightmost version component to increment:

  • ~> 3.0 means >= 3.0.0 and < 4.0.0
  • ~> 3.1.0 means >= 3.1.0 and < 3.2.0
  • Useful for allowing patches while preventing breaking changes
  • Recommended for production provider versions
