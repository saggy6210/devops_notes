Remote State & Backends

Learn to configure remote backends for secure, collaborative state management. Master S3, Azure, GCS backends and state sharing between configurations.

Why Remote Backends?

Collaboration

Team members share the same state

Locking

Prevents concurrent modifications

Security

Encryption at rest and in transit

Versioning

State history for rollback

S3 Backend (AWS)

1. Create Backend Infrastructure

S3 Bucket & DynamoDB 
# Create S3 bucket for state
aws s3api create-bucket \
  --bucket mycompany-terraform-state \
  --region us-east-1

# Enable versioning
aws s3api put-bucket-versioning \
  --bucket mycompany-terraform-state \
  --versioning-configuration Status=Enabled

# Enable encryption
aws s3api put-bucket-encryption \
  --bucket mycompany-terraform-state \
  --server-side-encryption-configuration '{
    "Rules": [{"ApplyServerSideEncryptionByDefault": {"SSEAlgorithm": "aws:kms"}}]
  }'

# Create DynamoDB table for locking
aws dynamodb create-table \
  --table-name terraform-locks \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

2. Configure Backend

backend.tf 
terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state"
    key            = "prod/network/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"
    
    # Optional: assume role for cross-account
    # role_arn = "arn:aws:iam::123456789:role/terraform"
  }
}

3. Initialize Backend

Migrate State 
# Initialize and migrate existing state
terraform init

# If migrating from local to remote:
# Terraform will ask: "Do you want to copy existing state?"
# Answer: yes

# Reconfigure backend
terraform init -reconfigure

# Migrate to new backend
terraform init -migrate-state

Azure Backend

Azure Storage Backend 
terraform {
  backend "azurerm" {
    resource_group_name  = "terraform-state-rg"
    storage_account_name = "tfstateaccount"
    container_name       = "tfstate"
    key                  = "prod.terraform.tfstate"
    
    # Use managed identity or service principal
    use_azuread_auth = true
  }
}

GCS Backend (Google Cloud)

GCS Backend 
terraform {
  backend "gcs" {
    bucket = "my-terraform-state"
    prefix = "prod/network"
  }
}

Sharing State Between Configurations

Use terraform_remote_state data source to access outputs from another configuration.

Remote State Data Source 
# In application configuration, reference network state
data "terraform_remote_state" "network" {
  backend = "s3"
  
  config = {
    bucket = "mycompany-terraform-state"
    key    = "prod/network/terraform.tfstate"
    region = "us-east-1"
  }
}

# Use outputs from network state
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = "t3.micro"
  subnet_id     = data.terraform_remote_state.network.outputs.private_subnet_ids[0]
  
  vpc_security_group_ids = [
    data.terraform_remote_state.network.outputs.app_security_group_id
  ]
}

Alternative: Use Data Sources

Instead of terraform_remote_state, consider using provider data sources (e.g., aws_vpc, aws_subnet) for looser coupling between configurations.

Backend Configuration Options

Partial Configuration

Partial Backend Config 
# backend.tf - partial configuration
terraform {
  backend "s3" {
    # Only specify immutable settings
    bucket = "mycompany-terraform-state"
    region = "us-east-1"
    # key will be provided at init
  }
}

# Initialize with remaining config
terraform init \
  -backend-config="key=prod/app/terraform.tfstate" \
  -backend-config="dynamodb_table=terraform-locks"

# Or use a backend config file
terraform init -backend-config=backend.hcl
backend.hcl 
key            = "prod/app/terraform.tfstate"
dynamodb_table = "terraform-locks"
encrypt        = true

Interview Q&A

Q1: Why use remote backends?
  • Team collaboration with shared state
  • State locking prevents corruption
  • Encryption for sensitive data
  • Versioning for rollback
  • Not stored in version control
Q2: How does state locking work?

State locking prevents concurrent operations:

  • S3: Uses DynamoDB for locks
  • Azure: Uses blob lease mechanism
  • GCS: Built-in locking
  • Lock acquired before plan/apply
  • Released after operation completes
Q3: How do you migrate state to a new backend?
  • Update backend configuration
  • Run terraform init -migrate-state
  • Confirm state migration when prompted
  • Verify state with terraform state list
Q4: What is terraform_remote_state?

Data source to read outputs from another state:

  • Enables state sharing between configs
  • Access via data.terraform_remote_state.name.outputs.value
  • Creates loose coupling between stacks
  • Alternative to hardcoding values
Q5: What is partial backend configuration?
  • Define some backend config in code
  • Supply rest at init time
  • Use -backend-config flag or .hcl file
  • Useful for CI/CD pipelines
  • Keeps secrets out of code
