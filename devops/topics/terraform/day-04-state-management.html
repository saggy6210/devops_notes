State Management

💾 Terraform State Management

Master Terraform state - the heart of how Terraform tracks and manages your infrastructure. Learn state commands, imports, and best practices.

What is Terraform State?

Terraform state is a JSON file that maps your configuration to real-world resources. It's the source of truth for Terraform.

Resource Mapping

Maps resource addresses to real infrastructure IDs (e.g., aws_instance.web → i-1234567890)

Performance

Caches resource attributes to avoid querying the provider on every plan

Dependencies

Tracks resource dependencies for proper creation/destruction order

Drift Detection

Compares actual infrastructure to desired state to detect changes

Important

State files contain sensitive data (passwords, keys). Never commit to version control. Use remote backends with encryption for production.

State File Structure

terraform.tfstate (simplified) 
{
  "version": 4,
  "terraform_version": "1.6.0",
  "serial": 5,
  "lineage": "abc123-def456-...",
  "outputs": {
    "instance_ip": {
      "value": "54.123.45.67",
      "type": "string"
    }
  },
  "resources": [
    {
      "mode": "managed",
      "type": "aws_instance",
      "name": "web",
      "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]",
      "instances": [
        {
          "attributes": {
            "id": "i-0abc123def456",
            "ami": "ami-12345678",
            "instance_type": "t3.micro",
            "public_ip": "54.123.45.67"
          }
        }
      ]
    }
  ]
}
FieldPurpose
versionState file format version
serialIncrements with each state change
lineageUnique ID for state history
outputsExported output values
resourcesAll managed resources and their attributes

State Commands

List Resources

terraform state list 
# List all resources in state
terraform state list

# Output:
# aws_instance.web
# aws_security_group.web_sg
# aws_vpc.main

# Filter by resource type
terraform state list aws_instance.*

Show Resource Details

terraform state show 
# Show specific resource
terraform state show aws_instance.web

# Output:
# resource "aws_instance" "web" {
#     ami                    = "ami-12345678"
#     id                     = "i-0abc123def456"
#     instance_type          = "t3.micro"
#     public_ip              = "54.123.45.67"
#     ...
# }

Move Resources

terraform state mv 
# Rename a resource (change name in config first)
terraform state mv aws_instance.web aws_instance.web_server

# Move to module
terraform state mv aws_instance.web module.compute.aws_instance.web

# Move from module to root
terraform state mv module.compute.aws_instance.web aws_instance.web

# Move between modules
terraform state mv module.old.aws_instance.web module.new.aws_instance.web

Remove from State

terraform state rm 
# Remove resource from state (doesn't delete actual resource)
terraform state rm aws_instance.web

# Remove module
terraform state rm module.vpc

# Dry run
terraform state rm -dry-run aws_instance.web

Pull and Push State

terraform state pull/push 
# Download remote state to stdout
terraform state pull > backup.tfstate

# Upload local state to remote backend
terraform state push backup.tfstate

# Force push (dangerous!)
terraform state push -force backup.tfstate

Importing Existing Resources

Import existing infrastructure into Terraform management using terraform import.

Traditional Import

terraform import 
# Step 1: Write resource block in config
resource "aws_instance" "imported" {
  # Attributes will be filled after import
}

# Step 2: Import the resource
terraform import aws_instance.imported i-0abc123def456

# Step 3: Run terraform plan to see current attributes
terraform plan

# Step 4: Update config to match actual resource
# Copy attributes from plan output to your config

Import Block (Terraform 1.5+)

import block 
# imports.tf
import {
  to = aws_instance.imported
  id = "i-0abc123def456"
}

import {
  to = aws_s3_bucket.data
  id = "my-existing-bucket"
}

# Run plan to generate config
terraform plan -generate-config-out=generated.tf

# Review generated.tf and copy to main config
# Remove import blocks after successful import

Common Import IDs

ResourceImport ID Format
aws_instanceInstance ID: i-1234567890

State Refresh and Replace

Refresh and Replace Commands 
# Refresh state to match actual infrastructure
terraform apply -refresh-only

# Force recreation of a resource
terraform apply -replace=aws_instance.web

# Multiple replacements
terraform apply -replace=aws_instance.web -replace=aws_instance.app

# Legacy taint (marks for recreation)
terraform taint aws_instance.web
terraform untaint aws_instance.web

State Locking

State locking prevents concurrent modifications that could corrupt state.

Purpose

Prevents multiple users from modifying state simultaneously

Automatic

Enabled by default on supported backends (S3+DynamoDB, Azure, GCS)

Lock Management 
# Disable locking (not recommended)
terraform apply -lock=false

# Set lock timeout
terraform apply -lock-timeout=5m

# Force unlock (emergency only!)
terraform force-unlock LOCK_ID

State Recovery

Recovery Scenarios 
# Backup before risky operations
terraform state pull > backup-$(date +%Y%m%d).tfstate

# Recover from backup
terraform state push backup-20240115.tfstate

# Restore from S3 versioning (if enabled)
aws s3api get-object \
  --bucket my-tf-state \
  --key terraform.tfstate \
  --version-id "abc123" \
  recovered.tfstate

# Re-import if state is lost
# 1. List resources manually from cloud console
# 2. Write resource blocks
# 3. Import each resource
terraform import aws_instance.web i-123456

Best Practices

Always enable versioning on your remote state bucket. Take backups before state manipulation operations.

Interview Q&A

Q1: What is Terraform state and why is it important?

Terraform state is a JSON file that maps configuration to real infrastructure:

  • Tracks resource IDs and metadata
  • Enables drift detection
  • Stores dependency information
  • Caches attributes for performance
  • Essential for plan/apply operations
Q2: How do you import existing infrastructure into Terraform?
  • Write resource block in configuration
  • Run terraform import resource.name resource_id
  • Run terraform plan to see current attributes
  • Update config to match actual state
  • Terraform 1.5+: Use import blocks for declarative import
Q3: What is state locking and why is it needed?

State locking prevents concurrent state modifications:

  • Avoids state corruption from parallel runs
  • Enabled by default on remote backends
  • Uses DynamoDB for S3, blob lease for Azure
  • Can be overridden with -lock=false (not recommended)
Q4: How do you move a resource in Terraform state?

Use terraform state mv:

  • Update resource name in config first
  • terraform state mv old_name new_name
  • Can move between modules
  • Run plan to verify no changes
Q5: What happens if you delete the state file?

Deleting state file has serious consequences:

  • Terraform loses track of all resources
  • Apply would try to create everything again
  • Results in duplicate resources or errors
  • Recovery: Re-import all resources manually
  • Prevention: Use remote backend with versioning
