🎯 Terraform Interview Deep Dive

Comprehensive interview preparation covering fundamental concepts, advanced topics, troubleshooting scenarios, and real-world architecture questions.

Fundamentals

What is Terraform and how does it differ from Ansible/CloudFormation?

Terraform is an IaC tool for provisioning infrastructure declaratively.

  • vs Ansible: Terraform provisions infra; Ansible configures servers. Terraform is declarative; Ansible is procedural. They're often used together.
  • vs CloudFormation: Terraform is cloud-agnostic (AWS, Azure, GCP); CF is AWS-only. Terraform uses HCL; CF uses JSON/YAML.
  • Key difference: Terraform maintains state and can detect drift across any provider.
Explain the Terraform workflow (init, plan, apply, destroy)
  • init: Initialize directory, download providers/modules, configure backend
  • plan: Create execution plan showing what will change (read-only)
  • apply: Execute plan to create/modify/delete resources
  • destroy: Delete all managed resources

Additional commands: validate (syntax check), fmt (format), refresh (sync state)

What is Terraform state and why is it important?

State is a JSON file mapping config to real infrastructure:

  • Tracks resource IDs and metadata
  • Enables drift detection
  • Determines create/update/delete actions
  • Caches attributes for performance
  • Contains sensitive data - secure it!
What are providers in Terraform?

Providers are plugins that enable Terraform to interact with APIs:

  • Each provider adds resource types and data sources
  • Examples: aws, azurerm, google, kubernetes
  • Configured in required_providers block
  • Downloaded during terraform init
  • Should be version-pinned in production

State Management

How do you configure remote state in S3? 
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"
  }
}

Requirements: S3 bucket with versioning, DynamoDB table for locking, appropriate IAM permissions.

What is state locking and why is it needed?

State locking prevents concurrent operations that could corrupt state:

  • Acquired before plan/apply
  • S3 uses DynamoDB for locks
  • Azure uses blob leases
  • Force unlock: terraform force-unlock LOCK_ID
How do you import existing resources into Terraform?
  • Write resource block in config
  • Run terraform import resource.name resource_id
  • Run plan to see actual attributes
  • Update config to match
  • Terraform 1.5+: Use import blocks for declarative import
How do you move resources between states?
  • terraform state mv - move within same state
  • terraform state rm + import - move between states
  • Use moved blocks for refactoring (Terraform 1.1+)

Modules & Variables

What are Terraform modules and their benefits?

Modules are reusable containers for resources:

  • Reusability: Write once, use many times
  • Encapsulation: Hide complexity
  • Standardization: Enforce conventions
  • Versioning: Control updates

Sources: Local, Registry, Git, S3, HTTP

Explain variable precedence in Terraform

Highest to lowest precedence:

  • -var and -var-file flags
  • *.auto.tfvars (alphabetically)
  • terraform.tfvars
  • Environment variables (TF_VAR_name)
  • Default values in variable definition
What is the difference between variables and locals?
  • Variables (var.): Input from users, can be set externally
  • Locals (local.): Computed internally, not exposed

Use locals to avoid repetition and simplify complex expressions.

Advanced Concepts

When would you use count vs for_each?
  • count: Identical resources, numeric index, reordering causes recreation
  • for_each: String keys, stable references, safe for additions/removals

Prefer for_each when resources have meaningful identifiers.

What are dynamic blocks?

Dynamic blocks generate repeated nested configuration:

dynamic "ingress" {
  for_each = var.ports
  content {
    from_port = ingress.value
    to_port   = ingress.value
    protocol  = "tcp"
  }
}
How do you share data between Terraform configurations?
  • terraform_remote_state data source
  • Data sources (aws_vpc, aws_subnet)
  • SSM Parameter Store / Secrets Manager
  • Output values from modules
Explain Terraform workspaces

Workspaces allow multiple state instances for same config:

  • Each workspace has its own state file
  • Access via terraform.workspace
  • Useful for dev/staging/prod of similar environments
  • Not recommended for significantly different configurations

Troubleshooting Scenarios

Terraform is trying to recreate a resource. How do you debug?
  • Check plan output for "forces replacement"
  • Review state: terraform state show resource
  • Check for immutable attributes (AMI, availability zone)
  • Use lifecycle { ignore_changes } if needed
  • Review count/for_each index changes
State file is corrupted. What do you do?
  • Restore from S3 versioning
  • Restore from backup
  • Recreate state by importing resources
  • Use terraform state push to restore

Prevention: Enable versioning, take regular backups before state operations.

How do you handle drift between actual infrastructure and state?
  • terraform plan shows drift
  • terraform apply -refresh-only to update state
  • If manual change should persist: update config to match
  • If manual change should revert: run apply
  • Prevention: Use ignore_changes for auto-managed attributes
"Error acquiring state lock" - how to resolve?
  • Wait for other operation to complete
  • Check if previous run crashed
  • Use terraform force-unlock LOCK_ID
  • Check DynamoDB/blob lease manually

Warning: Only force-unlock if certain no other operation is running.

Architecture & Best Practices

How would you structure Terraform for a multi-environment setup?
  • Shared modules: Create modules/ directory
  • Environment directories: environments/dev, staging, prod
  • Separate state files: Different S3 keys per environment
  • Variable files: env-specific .tfvars
  • CI/CD: Different pipelines per environment
How do you manage secrets in Terraform?
  • Never hardcode secrets in code
  • Use TF_VAR_ environment variables in CI
  • Integrate AWS Secrets Manager / HashiCorp Vault
  • Mark variables as sensitive = true
  • Encrypt remote state backend
Describe a CI/CD pipeline for Terraform
  • PR: fmt check, validate, tflint, security scan
  • PR: terraform plan, post as comment
  • Merge: terraform apply with auto-approve
  • Approvals: Manual approval for production
  • Tools: GitHub Actions, GitLab CI, Terraform Cloud
What are Terraform best practices for production?
  • Use remote backend with locking and encryption
  • Pin provider and module versions
  • Use modules for reusability
  • Keep state files small (layered architecture)
  • Run plan before apply, review changes
  • Use pre-commit hooks (fmt, validate, tflint)
  • Document with README and variable descriptions
  • Implement CI/CD with manual approval for prod

Quick Reference Commands

Essential Commands 
# Initialize
terraform init
terraform init -upgrade
terraform init -reconfigure

# Plan & Apply
terraform plan
terraform plan -out=tfplan
terraform apply tfplan
terraform apply -auto-approve
terraform apply -target=aws_instance.web

# State Operations
terraform state list
terraform state show aws_instance.web
terraform state mv old_name new_name
terraform state rm resource.name
terraform import resource.name resource_id

# Destroy
terraform destroy
terraform destroy -target=aws_instance.web

# Workspace
terraform workspace list
terraform workspace new dev
terraform workspace select prod

# Debug
TF_LOG=DEBUG terraform plan
terraform console
terraform output -json
