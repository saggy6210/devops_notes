Docker Security Best Practices

Harden containers, scan for vulnerabilities, and implement defense-in-depth strategies for production environments.

25-30 minReading time
5 Q&AInterview prep
SecurityCritical topic

Container Security Checklist

Docker containers share the host kernel. A container escape vulnerability can compromise the entire host. Security is critical!

  • Run as non-root user

    Never run containers as root. Use USER instruction in Dockerfile.

  • Use minimal base images

    Alpine, Distroless - fewer packages = smaller attack surface.

  • Scan images for vulnerabilities

    Use Trivy, Snyk, or Docker Scout in CI/CD pipeline.

  • Use read-only filesystem

    Containers shouldn't need to write to most directories.

  • Don't run with --privileged

    Privileged containers have full host access - avoid at all costs.

Running as Non-Root

Dockerfile with non-root user Secure 
FROM node:18-alpine

# Create app directory
WORKDIR /app

# Create non-root user
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001 -G nodejs

# Copy with correct ownership
COPY --chown=nodejs:nodejs . .

# Install dependencies
RUN npm ci --only=production

# Switch to non-root user
USER nodejs

EXPOSE 3000
CMD ["node", "server.js"]

Verify: docker run myapp whoami should output nodejs, not root

Vulnerability Scanning

Trivy - Popular Open Source Scanner 
# Install Trivy
brew install trivy  # or apt-get install trivy

# Scan an image
trivy image nginx:latest

# Scan and fail on HIGH/CRITICAL
trivy image --severity HIGH,CRITICAL --exit-code 1 myapp:latest

# Scan in CI/CD (GitHub Actions)
- name: Run Trivy vulnerability scanner
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: 'myapp:${{ github.sha }}'
    exit-code: '1'
    severity: 'CRITICAL,HIGH'

Other Scanning Tools

Docker Scout

Built into Docker Desktop

docker scout cves myapp:latest

Snyk

SaaS with CI integration

snyk container test myapp:latest

Runtime Security

Secure container run options 
# Read-only filesystem
docker run --read-only myapp

# Drop all capabilities, add only what's needed
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE myapp

# No new privileges escalation
docker run --security-opt=no-new-privileges myapp

# Limit resources
docker run --memory=512m --cpus=1 myapp

# Full secure example
docker run -d \
  --read-only \
  --cap-drop=ALL \
  --security-opt=no-new-privileges:true \
  --user 1000:1000 \
  --memory=256m \
  --pids-limit=50 \
  myapp:latest

Common Security Threats

Container Escape

Breaking out of container to host

Never use --privileged

Vulnerable Base Image

CVEs in outdated packages

Scan images in CI/CD

Secrets in Image

API keys, passwords in layers

Use secrets management

Root User

Running process as root

USER directive in Dockerfile

Interview Questions

1

How do you secure Docker containers in production?

  1. Run as non-root user (USER directive)
  2. Use minimal base images (Alpine, Distroless)
  3. Scan images for CVEs (Trivy, Snyk)
  4. Use read-only filesystems when possible
  5. Drop unnecessary Linux capabilities
  6. Limit container resources (memory, CPU)
  7. Use secrets management (not env vars for sensitive data)
2

What are Docker capabilities?

Linux capabilities divide root privileges into distinct units:

  • CAP_NET_BIND_SERVICE: Bind to ports below 1024
  • CAP_SYS_ADMIN: Broad system admin operations
  • CAP_CHOWN: Change file ownership

Best practice: --cap-drop=ALL then add only what's needed.

3

What is the risk of running containers as root?

  • Container escape gives root access to host
  • Malicious code can modify system files
  • Violates principle of least privilege
  • Many CVEs specifically target root containers
85% of container images run as root by default. Always add USER instruction!
4

How do you manage secrets in containers?

Never store secrets in:

  • Dockerfile (visible in layers)
  • Image environment variables
  • Version control

Instead use:

  • Docker Secrets (Swarm/Compose)
  • HashiCorp Vault
  • AWS Secrets Manager / Azure Key Vault
  • Kubernetes Secrets + External Secrets Operator
5

What is --privileged and why avoid it?

--privileged gives the container:

  • All Linux capabilities
  • Access to all host devices
  • Ability to modify host kernel
  • Essentially root on the host machine
If you need privileged mode, your architecture has a security flaw. Redesign instead.

Quick Tips

Enable Docker Content Trust: export DOCKER_CONTENT_TRUST=1

Avoid latest tag - pin to specific versions

Use internal networks, don't expose ports unnecessarily

Rebuild images regularly to get security patches
