What is Docker?

Docker is a platform that packages applications and their dependencies into containers — lightweight, portable units that run consistently across any environment.

Think of it like this:

A shipping container can carry any goods and be transported by ship, truck, or train. Similarly, a Docker container packages your application so it runs the same way everywhere — your laptop, staging, or production.

Docker Architecture

Docker uses a client-server architecture. Understanding this is crucial for interviews.

Docker Client

The CLI tool you use to interact with Docker. Commands like docker run, docker build are sent to the daemon.

Docker Daemon

The background service (dockerd) that manages Docker objects — images, containers, networks, and volumes.

Registry

Stores Docker images. Docker Hub is the default public registry. Private options include ECR, ACR, GCR.

Key Concepts

Docker Image

A read-only template with instructions to create a container. Built from a Dockerfile and stored in registries.

docker pull nginx:latest

Container

A running instance of an image. Has its own filesystem, network, and isolated process tree. Ephemeral by default.

docker run -d nginx

Dockerfile

A text file with instructions to build an image. Each instruction creates a layer in the final image.

docker build -t myapp .

Registry

A storage and distribution system for Docker images. Docker Hub is the default; private options include ECR, ACR, GCR.

docker push myrepo/myapp:v1

Essential Commands

Master these commands — they're used daily and asked in interviews.

Working with Images

docker pull nginx

Download an image from registry

docker images

List all local images

docker rmi nginx

Remove an image

Running Containers

docker run -d -p 80:80 --name web nginx

Run detached, map port 80, name it "web"

docker run -it ubuntu /bin/bash

Run interactive with terminal access

docker run -e MY_VAR=value nginx

Run with environment variable

Managing Containers

docker ps

List running containers

docker ps -a

List all containers (including stopped)

docker logs -f container_id

Follow container logs

docker exec -it container_id /bin/sh

Execute command inside running container

docker stop container_id

Stop a running container

docker rm container_id

Remove a stopped container

Interview Questions

Commonly asked questions with clear, concise answers.

Q1

What is the difference between Docker image and container?

An image is a read-only template (like a class in OOP), while a container is a running instance of that image (like an object).

  • Image: Blueprint with application code, runtime, libraries, and dependencies
  • Container: Running process created from an image with its own writable layer
You can create multiple containers from one image, each running independently.
Q2

Explain Docker architecture.

Docker uses a client-server architecture with three main components:

  1. Docker Client: CLI tool that sends commands via REST API
  2. Docker Daemon: Background service (dockerd) that manages containers, images, networks, volumes
  3. Docker Registry: Stores images (Docker Hub is the default)
Client Daemon Registry
Q3

Why use containers over VMs?

  • Faster startup: Seconds vs minutes
  • Lightweight: Share host OS kernel (MBs vs GBs)
  • Portable: Runs consistently across environments
  • Resource efficient: Higher density, less overhead
  • Microservices-friendly: One process per container
VMs still have their place: stronger isolation, different OS requirements, legacy applications.
Q4

What is the Docker daemon?

The Docker daemon (dockerd) is the background service that:

  • Listens for Docker API requests
  • Manages Docker objects (images, containers, networks, volumes)
  • Communicates with other daemons to manage services
systemctl status docker # Check daemon status
Q5

What happens when you run docker run nginx?

  1. Docker checks for nginx image locally
  2. If not found, pulls from Docker Hub
  3. Creates a new container from the image
  4. Allocates a read-write filesystem layer
  5. Creates network interface and assigns IP
  6. Starts the container and runs the default command
Use docker run --help to see all available options!

Scenario-Based Questions

Real-world problems to test practical knowledge.

Scenario 1: Container exits immediately after starting

Problem: You run docker run myapp but the container exits immediately.

Debugging steps:

  1. Check logs: docker logs container_id
  2. Run interactively: docker run -it myapp /bin/sh
  3. Check exit code: docker inspect container_id | grep ExitCode

Common causes:

  • Main process exits or crashes
  • Missing environment variables
  • Incorrect entrypoint or CMD
  • Permission issues

Scenario 2: How to reduce Docker image size?

Problem: Your Docker image is 1.5GB and takes too long to deploy.

Solutions:

1
Use multi-stage builds

Separate build and runtime stages

2
Use Alpine base images

~5MB vs ~100MB for standard images

3
Combine RUN commands

Fewer layers = smaller image

4
Use .dockerignore

Exclude node_modules, .git, etc.

5
Clean up in same layer

apt-get clean && rm -rf /var/lib/apt/lists/*

Quick Tips

Always use specific image tags, not :latest

Run containers as non-root user for security

Containers should be ephemeral and disposable

Use .dockerignore to reduce build context
