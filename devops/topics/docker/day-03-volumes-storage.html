Docker Volumes & Storage

Master data persistence in Docker. Learn volumes, bind mounts, tmpfs, and storage drivers for production-ready containers.

20-25 min Reading time
5 Q&A Interview prep
3 Scenarios Real-world

Why Data Persistence Matters

Containers are ephemeral — when a container is deleted, all data inside is lost. Docker provides storage options to persist data beyond the container lifecycle.

💡

Think of it like a hotel room

A container is like a hotel room — when you check out, the room is cleaned (data is gone). Volumes are like safety deposit boxes at the hotel — your valuables (data) persist even after checkout.

Docker Storage Types

Docker provides three ways to persist data. Understanding the differences is crucial for interviews!

Volumes

Managed by Docker. Best for persistent data. Stored in /var/lib/docker/volumes/

docker run -v mydata:/app/data nginx

Bind Mounts

Map host directory to container. Great for development. Full host path required.

docker run -v /host/path:/container/path nginx

tmpfs Mounts

Stored in memory only. Fast but not persistent. Good for sensitive data.

docker run --tmpfs /app/cache nginx

Essential Volume Commands

Create & Manage Volumes

docker volume create mydata Create a named volume
docker volume ls List all volumes
docker volume inspect mydata View volume details
docker volume rm mydata Remove a volume
docker volume prune Remove all unused volumes

Using Volumes with Containers

docker run -v mydata:/app/data nginx Mount named volume
docker run -v $(pwd):/app nginx Bind mount current directory
docker run -v mydata:/app:ro nginx Read-only volume mount
docker run --mount type=volume,src=mydata,dst=/app nginx Mount with explicit syntax

Interview Questions

1

What's the difference between volumes and bind mounts?

  • Volumes: Managed by Docker, stored in Docker area (/var/lib/docker/volumes/), portable, supports volume drivers
  • Bind Mounts: Maps host directory directly, depends on host filesystem structure, changes reflect immediately
Use volumes for production (portable) and bind mounts for development (live reload).
2

How do you persist database data in Docker?

Use named volumes to persist database data:

  • Create volume: docker volume create pgdata
  • Run with volume: docker run -v pgdata:/var/lib/postgresql/data postgres
  • Data persists even when container is removed
Never use anonymous volumes for important data - they're harder to manage and backup.
3

What is a Docker storage driver?

Storage drivers control how images and containers are stored on disk:

  • overlay2: Default, recommended for most use cases
  • devicemapper: For older systems without overlay support
  • btrfs/zfs: For specific filesystem features

Check current driver: docker info | grep "Storage Driver"

4

How do you share volumes between containers?

Multiple containers can mount the same volume:

  1. Create shared volume: docker volume create shared
  2. Container 1: docker run -v shared:/data app1
  3. Container 2: docker run -v shared:/data app2
Use this pattern for data sharing between microservices or log aggregation.
5

What happens to volumes when you run docker system prune?

docker system prune removes:

  • Stopped containers
  • Unused networks
  • Dangling images
  • NOT volumes by default!
To also remove volumes, use docker system prune --volumes or docker volume prune

Scenario-Based Questions

Database container losing data after restart

🔍 Problem

MySQL container loses all data when restarted or recreated.

✅ Solution
  1. Create named volume: docker volume create mysql_data
  2. Run with volume: docker run -v mysql_data:/var/lib/mysql mysql
  3. Data now persists across container restarts

Hot-reload for development environment

🎯 Goal

Code changes should reflect immediately without rebuilding the image.

✅ Solution - Use Bind Mount
  • docker run -v $(pwd)/src:/app/src node-app
  • Changes to local src/ folder appear instantly in container
  • Combine with nodemon/webpack-dev-server for auto-restart

Storing secrets temporarily in container

🎯 Requirement

Store sensitive data that shouldn't be persisted to disk.

✅ Solution - Use tmpfs
  • docker run --tmpfs /run/secrets:rw,noexec,nosuid,size=64m myapp
  • Data exists only in memory
  • Automatically cleared when container stops

Quick Tips

Use named volumes over anonymous volumes for easier management

Use :ro suffix for read-only mounts when possible

Backup volumes: docker run --rm -v mydata:/data -v $(pwd):/backup alpine tar cvf /backup/data.tar /data

Find volume location: docker volume inspect mydata | grep Mountpoint
