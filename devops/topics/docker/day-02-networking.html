Docker Networking

Master Docker networking concepts. Learn about network drivers, bridge networks, container communication, and ace those interview questions.

20-25 min Reading time
6 Q&A Interview prep
2 Scenarios Real-world

What is Docker Networking?

Docker networking allows containers to communicate with each other, the Docker host, and external networks. Docker uses a pluggable network driver system with different drivers for different use cases.

💡

Think of it like a building's network

Bridge network is like an internal office network where departments (containers) can talk. Host network is like connecting directly to the building's main line. Overlay is like a VPN connecting multiple office buildings together.

Network Drivers

Docker provides several network drivers. Understanding these is crucial for interviews!

Bridge (Default)

Creates private internal network. Containers communicate by IP. User-defined bridges allow DNS resolution by name.

docker network create my-bridge

Host

Container shares host's network stack. No isolation, no port mapping needed. Best for performance-sensitive apps.

docker run --network host nginx

Overlay

Enables multi-host communication in Docker Swarm. Creates distributed network across nodes.

docker network create -d overlay my-overlay

Macvlan

Assigns MAC address to container. Appears as physical device on network. Useful for legacy apps.

docker network create -d macvlan my-macvlan

Essential Commands

docker network ls List all Docker networks
docker network create my-app-network Create user-defined bridge network
docker run -d --network my-app-network --name web nginx Run container on specific network
docker network connect my-network my-container Connect running container to network
docker network disconnect my-network my-container Disconnect container from network
docker network inspect my-network View network details, connected containers, IPs
docker run -d -p 8080:80 nginx Port mapping: host 8080 → container 80

Interview Questions

1

What are the different network drivers in Docker?

Docker provides these main network drivers:

  • Bridge - Default driver for standalone containers
  • Host - Removes network isolation, shares host networking
  • Overlay - Multi-host networking for Swarm services
  • Macvlan - Assigns MAC address, appears as physical device
  • None - Disables all networking
2

What's the difference between default bridge and user-defined bridge?

  • Default Bridge: Containers communicate by IP only, not by name. Less isolation.
  • User-defined Bridge: Containers can communicate by name (DNS resolution). Better isolation. Automatic DNS. Recommended for production.

Create user-defined: docker network create my-network

3

How do containers on the same network communicate?

On a user-defined bridge network, containers can communicate using:

  • Container name as hostname (e.g., ping database)
  • Container IP address
  • Network alias if specified

Docker's embedded DNS server handles name resolution automatically.

4

When would you use the host network driver?

Use host network when:

  • Performance is critical (no NAT overhead)
  • Container needs access to all host network interfaces
  • Running monitoring tools that need host network visibility
  • Exposing many ports (avoids port mapping complexity)

⚠️ Tradeoff: No network isolation between container and host.

5

Explain port mapping in Docker.

Port mapping exposes container ports to the host:

  • -p 8080:80 - Map host port 8080 to container port 80
  • -p 80 - Map random host port to container port 80
  • -p 127.0.0.1:8080:80 - Bind only to localhost
  • -P - Publish all exposed ports to random ports
6

What is the overlay network used for?

Overlay networks are used in Docker Swarm for:

  • Multi-host communication between containers
  • Connecting services across different Docker hosts
  • Creating distributed networks spanning multiple nodes
  • Enabling service discovery across the cluster

Create: docker network create -d overlay my-swarm-network

Scenario-Based Questions

Web app can't connect to database container by name

🔍 Problem

Two containers (web-app, database) on default bridge network. web-app cannot resolve "database" hostname.

✅ Solution
  1. Create user-defined network: docker network create app-network
  2. Run database: docker run -d --name database --network app-network postgres
  3. Run web-app: docker run -d --name web-app --network app-network -p 80:80 myapp

Result: web-app can now connect to database:5432

Design network for 3-tier application

📋 Requirements

Web (Nginx), App (Node.js), Database (PostgreSQL) on single host. Only Web should be accessible externally.

✅ Solution
  1. Create network: docker network create app-tier
  2. Database (no port exposed): docker run -d --name db --network app-tier postgres
  3. App: docker run -d --name app --network app-tier myapp
  4. Web (expose port): docker run -d --name web --network app-tier -p 80:80 nginx

Security: DB is isolated, only internal containers can access it via db:5432
