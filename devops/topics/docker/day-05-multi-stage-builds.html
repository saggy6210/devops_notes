Docker Multi-stage Builds

Create lean production images by separating build and runtime environments. Reduce image sizes by up to 90%.

20-25 min Reading time
5 Q&A Interview prep
90% smaller Image size

What are Multi-stage Builds?

Multi-stage builds use multiple FROM statements in a single Dockerfile. Each FROM starts a new stage. You copy only what you need from previous stages to the final image.

💡

Think of it like cooking

You use many tools and ingredients (build dependencies) to cook a meal, but you only serve the finished dish (final binary) to guests. Multi-stage builds let you discard the "kitchen mess" and deliver only the clean result.

Build Stage

Compilers, SDKs, dev deps

Copy Artifacts

Only built files

Production Stage

Minimal runtime only

Node.js Multi-stage Build

Dockerfile - Node.js App ~50MB final 
# Stage 1: Build
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production
FROM node:18-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY --from=builder /app/dist ./dist

USER node
EXPOSE 3000
CMD ["node", "dist/index.js"]

Without Multi-stage

~950MB

With Multi-stage

~50MB
↓ 95% reduction!

Go Multi-stage Build

Dockerfile - Go App (Scratch base) ~10MB final 
# Stage 1: Build
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o main .

# Stage 2: Minimal Production
FROM scratch
COPY --from=builder /app/main /main
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
ENTRYPOINT ["/main"]
Scratch is an empty base image — no OS, no shell, just your binary. Perfect for compiled languages like Go and Rust.

Java/Spring Boot Multi-stage Build

Dockerfile - Spring Boot with Maven ~200MB final 
# Stage 1: Build with Maven
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# Stage 2: Runtime only
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar

USER 1000
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Full JDK + Maven

~800MB

JRE Alpine Only

~200MB
↓ 75% reduction!

Interview Questions

1

What is a multi-stage Docker build?

Multi-stage builds allow you to use multiple FROM statements in a single Dockerfile:

  • Each FROM starts a new build stage
  • You can copy artifacts from one stage to another
  • Only the final stage becomes your image
  • Build tools and dependencies don't end up in production
2

Why use multi-stage builds?

  1. Smaller images: No build tools in final image
  2. Security: Fewer packages = smaller attack surface
  3. Single Dockerfile: No need for separate build/runtime files
  4. Faster deployments: Smaller images transfer faster
3

How do you copy files between stages?

Use COPY --from=stage_name or COPY --from=0 (stage number):

  • Named: COPY --from=builder /app/dist ./dist
  • Numbered: COPY --from=0 /app/binary ./
  • External: COPY --from=nginx:alpine /etc/nginx/nginx.conf ./
You can even copy from external images, not just previous stages!
4

What is "scratch" base image?

FROM scratch is an empty image with:

  • No operating system
  • No shell
  • No utilities
  • Just your binary
Only works with statically compiled binaries (Go, Rust). Not for interpreted languages.
5

How do you build a specific stage?

Use --target flag to build up to a specific stage:

docker build --target builder -t myapp:builder .

Useful for:

  • Running tests in CI without building production image
  • Debugging build issues
  • Creating development-specific images

Quick Tips

Name stages with AS name for readability

Use distroless images: gcr.io/distroless/base

Check final size: docker images myapp

Add test stage: FROM builder AS test
