Docker in CI/CD Pipeline

In modern DevOps, Docker is central to CI/CD. Build once, deploy everywhere - the same container runs in dev, staging, and production.

Build Image
Scan & Test

GitHub Actions - Docker Build & Push 
name: Docker CI/CD

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Login to DockerHub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}
      
      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: myrepo/myapp:${{ github.sha }}
      
      - name: Scan for vulnerabilities
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myrepo/myapp:${{ github.sha }}
          exit-code: '1'
          severity: 'CRITICAL,HIGH'

Production Readiness Checklist

  • Health Checks

    Add HEALTHCHECK instruction or configure orchestrator health probes

  • Resource Limits

    Set memory and CPU limits to prevent resource starvation

  • Graceful Shutdown

    Handle SIGTERM signal for clean shutdown (connections, cleanup)

  • Log to stdout

    Application logs to stdout/stderr, configure log driver for aggregation

  • Non-root User

    Run containers as non-root for security

  • Immutable Tags

    Use SHA or semantic versioning, never :latest in production

Container Registries

Push to different registries 
# Docker Hub
docker push username/myapp:v1.0.0

# AWS ECR
aws ecr get-login-password | docker login --username AWS --password-stdin 123456789.dkr.ecr.us-east-1.amazonaws.com
docker tag myapp:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0.0
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0.0

# Azure ACR
az acr login --name myregistry
docker push myregistry.azurecr.io/myapp:v1.0.0

# Google GCR
gcloud auth configure-docker
docker push gcr.io/myproject/myapp:v1.0.0

Container Orchestration

For production workloads, use container orchestration: Kubernetes (most popular), Docker Swarm, or managed services like ECS/Fargate.

Kubernetes Deployment example 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myrepo/myapp:v1.0.0
        ports:
        - containerPort: 3000
        resources:
          limits:
            memory: "256Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 10
        readinessProbe:
          httpGet:
            path: /ready
            port: 3000

Interview Questions

1

How do you deploy Docker containers in production?

  1. Build image in CI/CD pipeline
  2. Push to private container registry
  3. Deploy using orchestrator (Kubernetes, ECS)
  4. Use rolling updates for zero downtime
  5. Monitor with health checks and logging
2

What is a container registry?

A storage and distribution service for container images:

  • Docker Hub: Default public registry
  • AWS ECR: Amazon's private registry
  • Azure ACR: Azure Container Registry
  • GCR/Artifact Registry: Google Cloud
  • Harbor: Self-hosted enterprise option
3

What is zero-downtime deployment?

Deployment strategies that don't cause service interruption:

  • Rolling update: Gradually replace old with new
  • Blue-green: Two identical environments, switch traffic
  • Canary: Route small % to new version first
Health checks are essential - don't route traffic until new containers are healthy!
4

Why use Kubernetes over Docker Swarm?

Kubernetes advantages:

  • Industry standard, massive ecosystem
  • Better auto-scaling capabilities
  • More sophisticated networking (Ingress)
  • ConfigMaps, Secrets, RBAC
  • Available as managed service (EKS, AKS, GKE)

Swarm is simpler but less feature-rich.

5

How do you handle configuration in containers?

  • Environment variables: Non-sensitive config
  • ConfigMaps (K8s): External configuration
  • Secrets: Sensitive data (encrypted)
  • Volume mounts: Config files from external source

12-factor app principle: Config should be in the environment, not baked into image.

Quick Tips

Tag with git SHA: myapp:abc123 for traceability

Always implement health endpoints: /health, /ready

Set restart policy: unless-stopped or always

Scan images in CI before pushing to registry
