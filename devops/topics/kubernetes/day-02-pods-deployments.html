Understanding Pods

A Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process and can contain one or more containers that share storage, network, and lifecycle.

Pod Characteristics

  • Shared Network - All containers share the same IP address and port space
  • Shared Storage - Volumes are accessible by all containers
  • Co-located - Containers are always scheduled on the same node
  • Ephemeral - Pods are mortal; they can be created and destroyed
Create a Pod kubectl run nginx --image=nginx:1.21 --port=80 # From YAML kubectl apply -f pod.yaml # Generate YAML without creating kubectl run nginx --image=nginx --dry-run=client -o yaml > pod.yaml
Pod YAML Example apiVersion: v1 kind: Pod metadata: name: multi-container-pod labels: app: myapp spec: containers: - name: main-app image: nginx:1.21 ports: - containerPort: 80 resources: requests: memory: "64Mi" cpu: "250m" limits: memory: "128Mi" cpu: "500m" - name: sidecar image: busybox command: ['sh', '-c', 'while true; do echo sidecar; sleep 10; done']

Pod Management Commands

View Pods kubectl get pods kubectl get pods -o wide # More details kubectl get pods --show-labels # Show labels kubectl get pods -l app=nginx # Filter by label kubectl get pods --field-selector=status.phase=Running
Debugging Pods kubectl describe pod <pod-name> kubectl logs <pod-name> kubectl logs <pod-name> -c <container> # Multi-container pod kubectl logs <pod-name> --previous # Previous container logs kubectl logs -f <pod-name> # Stream logs
Execute Commands kubectl exec -it <pod-name> -- /bin/sh kubectl exec <pod-name> -- ls /app kubectl exec <pod-name> -c <container> -- cat /etc/config
Port Forwarding & Copy kubectl port-forward pod/nginx 8080:80 kubectl cp <pod-name>:/path/file ./local-file kubectl cp ./local-file <pod-name>:/path/

Deployments

A Deployment provides declarative updates for Pods and ReplicaSets. It's the recommended way to manage stateless applications.

Key Features

  • Declarative Updates - Define desired state, K8s handles the rest
  • Rolling Updates - Zero-downtime deployments
  • Rollback - Easy revert to previous versions
  • Scaling - Manual and auto-scaling support
  • Self-healing - Replaces failed pods automatically
Deployment YAML apiVersion: apps/v1 kind: Deployment metadata: name: nginx-deployment labels: app: nginx spec: replicas: 3 selector: matchLabels: app: nginx strategy: type: RollingUpdate rollingUpdate: maxSurge: 1 maxUnavailable: 0 template: metadata: labels: app: nginx spec: containers: - name: nginx image: nginx:1.21 ports: - containerPort: 80 readinessProbe: httpGet: path: / port: 80 initialDelaySeconds: 5 periodSeconds: 10 livenessProbe: httpGet: path: / port: 80 initialDelaySeconds: 15 periodSeconds: 20

Deployment Operations

Create & Manage kubectl create deployment nginx --image=nginx:1.21 --replicas=3 kubectl apply -f deployment.yaml kubectl get deployments kubectl describe deployment nginx-deployment
Scaling kubectl scale deployment nginx --replicas=5 kubectl autoscale deployment nginx --min=3 --max=10 --cpu-percent=80
Image Updates & Rollouts kubectl set image deployment/nginx nginx=nginx:1.22 kubectl rollout status deployment/nginx kubectl rollout history deployment/nginx kubectl rollout undo deployment/nginx kubectl rollout undo deployment/nginx --to-revision=2
Pause & Resume kubectl rollout pause deployment/nginx kubectl set image deployment/nginx nginx=nginx:1.23 kubectl set resources deployment/nginx -c=nginx --limits=cpu=200m kubectl rollout resume deployment/nginx

Best Practice

Always use kubectl rollout pause when making multiple changes to avoid triggering multiple rollouts. Resume when all changes are ready.

Interview Q&A

Q: What is the difference between a Deployment and a ReplicaSet?

ReplicaSet ensures a specified number of pod replicas are running. It handles pod replication but doesn't support rolling updates.

Deployment is a higher-level abstraction that manages ReplicaSets and provides:

  • Declarative updates
  • Rolling updates and rollbacks
  • Version history
  • Pause/resume functionality

Always use Deployments for stateless apps; ReplicaSets are created automatically.

Q: What is the difference between liveness and readiness probes?

Liveness Probe: Determines if a container is running. If it fails, kubelet kills the container and restarts it based on restart policy.

Readiness Probe: Determines if a container is ready to serve traffic. If it fails, the pod is removed from Service endpoints but NOT restarted.

Startup Probe: Used for slow-starting containers. Disables liveness/readiness until it succeeds.

Q: What is maxSurge and maxUnavailable in rolling updates?

maxSurge: Maximum number of pods that can be created above the desired replica count during update. Can be absolute number or percentage.

maxUnavailable: Maximum number of pods that can be unavailable during update.

Example with 10 replicas, maxSurge=2, maxUnavailable=1:

  • Maximum pods during update: 12 (10 + 2)
  • Minimum available pods: 9 (10 - 1)
Q: When would you use a StatefulSet over a Deployment?

Use StatefulSet when you need:

  • Stable network identity - Pods get persistent hostname (web-0, web-1)
  • Stable storage - Each pod gets its own PVC that persists
  • Ordered deployment - Pods created/deleted in order
  • Ordered scaling - Scale up/down happens sequentially

Examples: Databases (MySQL, PostgreSQL), message queues (Kafka, RabbitMQ), distributed systems (Elasticsearch, Cassandra)

Q: How do you debug a pod stuck in CrashLoopBackOff?
  1. kubectl describe pod <name> - Check Events section
  2. kubectl logs <name> --previous - View crashed container logs
  3. Check image name and tag are correct
  4. Verify command/args in spec
  5. Check resource limits (OOMKilled)
  6. Verify ConfigMaps/Secrets exist
  7. kubectl get events --sort-by='.lastTimestamp'
