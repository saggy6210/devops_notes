Kubernetes Services

A Service is an abstraction that defines a logical set of Pods and a policy to access them. Services enable loose coupling between dependent Pods.

Why Services?

  • Pods are ephemeral - IPs change when pods restart
  • Load balancing across multiple pod replicas
  • Service discovery via DNS
  • Decoupling frontend from backend

ClusterIP (Default)

Exposes the Service on an internal IP. Only reachable from within the cluster. Best for internal communication.

NodePort

Exposes the Service on each Node's IP at a static port (30000-32767). Accessible from outside using <NodeIP>:<NodePort>.

LoadBalancer

Exposes the Service externally using a cloud provider's load balancer. Creates NodePort and ClusterIP automatically.

ExternalName

Maps the Service to a DNS name (e.g., external database). No proxying; returns a CNAME record.

Service Examples

ClusterIP Service apiVersion: v1 kind: Service metadata: name: my-service spec: type: ClusterIP selector: app: myapp ports: - protocol: TCP port: 80 # Service port targetPort: 8080 # Container port
NodePort Service apiVersion: v1 kind: Service metadata: name: my-nodeport spec: type: NodePort selector: app: myapp ports: - port: 80 targetPort: 8080 nodePort: 30080 # Optional: 30000-32767
LoadBalancer Service apiVersion: v1 kind: Service metadata: name: my-loadbalancer annotations: service.beta.kubernetes.io/aws-load-balancer-type: "nlb" spec: type: LoadBalancer selector: app: myapp ports: - port: 80 targetPort: 8080
Service Commands kubectl expose deployment nginx --port=80 --type=ClusterIP kubectl get services kubectl describe service my-service kubectl get endpoints my-service

Kubernetes DNS

Kubernetes creates DNS records for Services and Pods. CoreDNS runs as a cluster add-on.

DNS Resolution Patterns # Service DNS (within same namespace) my-service # Service DNS (cross-namespace) my-service.namespace # Full FQDN my-service.namespace.svc.cluster.local # Pod DNS pod-ip-address.namespace.pod.cluster.local

Best Practice

Use Service names for inter-service communication. Never hardcode Pod IPs as they change when pods restart.

Ingress

Ingress manages external access to services, typically HTTP/HTTPS. Provides load balancing, SSL termination, and name-based virtual hosting.

Basic Ingress apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: my-ingress annotations: nginx.ingress.kubernetes.io/rewrite-target: / spec: ingressClassName: nginx rules: - host: myapp.example.com http: paths: - path: / pathType: Prefix backend: service: name: my-service port: number: 80
Ingress with TLS apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: tls-ingress spec: ingressClassName: nginx tls: - hosts: - myapp.example.com secretName: tls-secret rules: - host: myapp.example.com http: paths: - path: / pathType: Prefix backend: service: name: my-service port: number: 80

Important

Ingress requires an Ingress Controller (nginx-ingress, traefik, HAProxy, etc.) to be installed in the cluster.

Network Policies

Network Policies control traffic flow at the IP/port level. By default, pods accept traffic from any source.

Deny All Ingress apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: deny-all-ingress spec: podSelector: {} policyTypes: - Ingress
Allow Specific Traffic apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-frontend spec: podSelector: matchLabels: app: backend policyTypes: - Ingress ingress: - from: - podSelector: matchLabels: app: frontend ports: - protocol: TCP port: 8080

Interview Q&A

Q: What is the difference between ClusterIP, NodePort, and LoadBalancer?

ClusterIP: Internal only, default type. Pods within cluster access via service name.

NodePort: Exposes service on static port on each node. External access via NodeIP:NodePort.

LoadBalancer: Provisions external LB from cloud provider. Best for production external access.

Each type builds on the previous: LoadBalancer creates NodePort which creates ClusterIP.

Q: What is the difference between Ingress and Service?

Service: L4 load balancing (TCP/UDP). Each service gets its own IP/port.

Ingress: L7 load balancing (HTTP/HTTPS). Single entry point with routing rules based on host/path.

Ingress can route to multiple services, provide SSL termination, and use one LoadBalancer for many services (cost-effective).

Q: How does Kubernetes service discovery work?

Two mechanisms:

  1. Environment Variables: When a pod runs, kubelet adds env vars for each active service.
  2. DNS: CoreDNS creates DNS records. Pods resolve service names to ClusterIP.

DNS is preferred as it works for services created after the pod.

Q: What are Endpoints in Kubernetes?

Endpoints are automatically created when you create a Service with a selector. They contain the IP addresses of pods matching the selector.

When a pod is added/removed, Endpoints are updated. kube-proxy uses Endpoints to route traffic.

You can create Endpoints manually for external services without selectors.

Q: What CNI plugins have you worked with?

Common CNI plugins:

  • Calico: Network policies, BGP routing, high performance
  • Flannel: Simple overlay network, good for beginners
  • Weave: Easy setup, encrypted traffic
  • Cilium: eBPF-based, advanced network policies
  • AWS VPC CNI: Native VPC networking for EKS
  • Azure CNI: Native Azure networking for AKS
