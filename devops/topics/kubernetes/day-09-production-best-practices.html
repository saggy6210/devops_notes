Resource Management

Resource Requests & Limits apiVersion: v1 kind: Pod metadata: name: app spec: containers: - name: app image: myapp:latest resources: requests: # Guaranteed resources memory: "256Mi" cpu: "250m" limits: # Maximum resources memory: "512Mi" cpu: "500m"

Critical

Always set resource requests and limits. Without them, a single pod can consume all node resources, affecting other workloads.

QoS Classes

  • Guaranteed - requests = limits for all containers
  • Burstable - At least one container has requests < limits
  • BestEffort - No requests or limits set (first to be evicted)

Production Security Checklist

  • Enable RBAC and limit cluster-admin access
  • Use namespace isolation
  • Apply Pod Security Standards (restricted)
  • Run containers as non-root user
  • Scan container images for vulnerabilities
  • Enable audit logging
  • Use Network Policies to restrict traffic
  • Encrypt secrets at rest
  • Use separate ServiceAccounts per application
  • Disable automountServiceAccountToken when not needed
  • Keep Kubernetes and images updated
  • Use private container registries

Reliability Checklist

  • Set resource requests and limits
  • Configure liveness and readiness probes
  • Use multiple replicas (minimum 3 for HA)
  • Configure Pod Disruption Budgets
  • Use anti-affinity for spreading across nodes/zones
  • Implement graceful shutdown (preStop hooks)
  • Set up HPA for automatic scaling
  • Configure Cluster Autoscaler for nodes
  • Use proper deployment strategies (rolling update)
  • Implement retry logic in applications
  • Set up monitoring and alerting
  • Have disaster recovery plan

Interview Q&A

Q: How do you ensure zero-downtime deployments?
  1. Use rolling update strategy with appropriate maxSurge/maxUnavailable
  2. Configure proper readiness probes
  3. Set minReadySeconds to ensure pod stability
  4. Implement graceful shutdown with preStop hooks
  5. Use Pod Disruption Budgets
  6. Have multiple replicas spread across nodes
Q: What happens when a node runs out of memory?

Kubelet triggers eviction based on QoS class:

  1. BestEffort pods evicted first
  2. Burstable pods exceeding requests evicted next
  3. Guaranteed pods evicted last (only if no others)

Prevention: Set appropriate resource limits, use LimitRanges, monitor resource usage.

Q: How do you handle database migrations in Kubernetes?

Common approaches:

  • Init Container - Run migrations before app starts
  • Helm pre-upgrade hook - Job that runs before deployment
  • Separate Job - Manual or CI/CD triggered migration job
  • Blue-Green - Migrate to new DB, switch traffic

Best practice: Make migrations backward compatible for zero-downtime deployments.

Q: How do you debug a pod that won't start?
  1. kubectl describe pod - Check Events section
  2. kubectl logs --previous - Crashed container logs
  3. kubectl get events --sort-by='.lastTimestamp'
  4. Check image pull issues (ImagePullBackOff)
  5. Verify ConfigMaps/Secrets exist
  6. Check resource availability on nodes
  7. Verify PVC binding status
  8. kubectl get pod -o yaml - Check spec and status
