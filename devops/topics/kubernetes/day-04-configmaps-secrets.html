Learn to manage application configuration and sensitive data in Kubernetes.

ConfigMaps

ConfigMaps store non-confidential configuration data as key-value pairs. They decouple configuration from container images, making applications portable.

Creating ConfigMaps

From Literal Values kubectl create configmap app-config \ --from-literal=DATABASE_HOST=mysql.default.svc \ --from-literal=DATABASE_PORT=3306
From File kubectl create configmap nginx-config --from-file=nginx.conf kubectl create configmap app-config --from-file=config/
YAML Definition apiVersion: v1 kind: ConfigMap metadata: name: app-config data: DATABASE_HOST: mysql.default.svc DATABASE_PORT: "3306" config.json: | { "logLevel": "info", "maxConnections": 100 }

Using ConfigMaps in Pods

As Environment Variables apiVersion: v1 kind: Pod metadata: name: app-pod spec: containers: - name: app image: myapp:latest envFrom: - configMapRef: name: app-config # Or individual keys: env: - name: DB_HOST valueFrom: configMapKeyRef: name: app-config key: DATABASE_HOST
As Volume Mount apiVersion: v1 kind: Pod metadata: name: app-pod spec: containers: - name: app image: myapp:latest volumeMounts: - name: config-volume mountPath: /etc/config volumes: - name: config-volume configMap: name: app-config items: - key: config.json path: app-config.json

Hot Reload

ConfigMaps mounted as volumes are updated automatically (with some delay). Environment variables require pod restart.

Secrets

Secrets store sensitive information like passwords, tokens, and keys. They're similar to ConfigMaps but designed for confidential data.

Secret Types

  • Opaque - Default type for arbitrary user data
  • kubernetes.io/service-account-token - Service account token
  • kubernetes.io/dockerconfigjson - Docker registry credentials
  • kubernetes.io/tls - TLS certificate and key
  • kubernetes.io/basic-auth - Basic authentication credentials
Create Secrets # From literal (base64 encoded automatically) kubectl create secret generic db-secret \ --from-literal=username=admin \ --from-literal=password=S3cr3t! # From file kubectl create secret generic ssh-key --from-file=ssh-privatekey=~/.ssh/id_rsa # TLS secret kubectl create secret tls tls-secret \ --cert=path/to/cert.pem \ --key=path/to/key.pem # Docker registry kubectl create secret docker-registry regcred \ --docker-server=https://index.docker.io/v1/ \ --docker-username=user \ --docker-password=pass
YAML Definition apiVersion: v1 kind: Secret metadata: name: db-secret type: Opaque data: username: YWRtaW4= # base64 encoded password: UzNjcjN0IQ== --- # Using stringData (plain text, encoded on creation) apiVersion: v1 kind: Secret metadata: name: db-secret type: Opaque stringData: username: admin password: S3cr3t!

Security Note

Secrets are base64 encoded, NOT encrypted by default. Enable encryption at rest and use RBAC to restrict access. Consider external secret managers (Vault, AWS Secrets Manager).

Using Secrets in Pods

As Environment Variables apiVersion: v1 kind: Pod metadata: name: app-pod spec: containers: - name: app image: myapp:latest env: - name: DB_USERNAME valueFrom: secretKeyRef: name: db-secret key: username - name: DB_PASSWORD valueFrom: secretKeyRef: name: db-secret key: password
As Volume Mount apiVersion: v1 kind: Pod metadata: name: app-pod spec: containers: - name: app image: myapp:latest volumeMounts: - name: secret-volume mountPath: /etc/secrets readOnly: true volumes: - name: secret-volume secret: secretName: db-secret defaultMode: 0400 # Read-only for owner
Image Pull Secret apiVersion: v1 kind: Pod metadata: name: private-app spec: containers: - name: app image: private-registry.io/myapp:latest imagePullSecrets: - name: regcred

ConfigMaps vs Secrets

ConfigMap

  • Non-sensitive configuration
  • Stored in plain text
  • 1MB size limit
  • Can use binaryData for binary files

Secret

  • Sensitive data
  • Base64 encoded (not encrypted)
  • 1MB size limit
  • Stored in tmpfs (memory) on nodes

Management Commands

ConfigMap Commands kubectl get configmaps kubectl describe configmap app-config kubectl get configmap app-config -o yaml kubectl edit configmap app-config kubectl delete configmap app-config
Secret Commands kubectl get secrets kubectl describe secret db-secret kubectl get secret db-secret -o yaml # Decode base64 secret kubectl get secret db-secret -o jsonpath='{.data.password}' | base64 -d

Interview Q&A

Q: Are Kubernetes Secrets secure?

By default, Secrets are only base64 encoded, NOT encrypted. For security:

  • Enable encryption at rest in etcd
  • Use RBAC to restrict Secret access
  • Enable audit logging
  • Consider external secret managers (HashiCorp Vault, AWS Secrets Manager)
  • Use sealed-secrets or SOPS for GitOps
Q: How do you update ConfigMaps/Secrets without restarting pods?

For volume-mounted ConfigMaps/Secrets, Kubernetes updates them automatically (with a delay up to kubelet sync period).

For environment variables, the pod must be restarted. Options:

  • Use volume mounts instead of env vars
  • Use a sidecar that watches for changes
  • Implement config reload in your application
  • Use Reloader controller (stakater/reloader)
Q: What is the size limit for ConfigMaps and Secrets?

Both ConfigMaps and Secrets have a 1MB limit. This is a limit of the etcd value size.

If you need larger configs, consider:

  • Mounting from external storage (NFS, S3)
  • Using init containers to fetch configs
  • Splitting into multiple ConfigMaps
Q: How do you manage secrets in GitOps workflows?

Since secrets shouldn't be stored in Git in plain text:

  • Sealed Secrets: Encrypt secrets that only the cluster can decrypt
  • SOPS: Mozilla's secret operations tool with KMS encryption
  • External Secrets Operator: Sync from Vault, AWS SM, etc.
  • Vault Agent Injector: HashiCorp Vault integration
  • ArgoCD Vault Plugin: Fetch secrets during deployment
