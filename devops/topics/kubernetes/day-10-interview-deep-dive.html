Architecture Questions

Q: Walk me through what happens when you run `kubectl apply -f deployment.yaml`
  1. kubectl reads YAML and sends HTTP POST to API server
  2. API Server authenticates, authorizes, validates request
  3. API Server writes Deployment object to etcd
  4. Deployment Controller watches for new Deployments, creates ReplicaSet
  5. ReplicaSet Controller creates Pod objects
  6. Scheduler assigns pods to nodes
  7. Kubelet on assigned node pulls image and starts container
  8. Kubelet reports pod status back to API server
Q: What is the difference between a control plane and worker node?

Control Plane (Master):

  • kube-apiserver - API endpoint
  • etcd - Cluster data store
  • kube-scheduler - Pod scheduling
  • kube-controller-manager - Controllers
  • cloud-controller-manager - Cloud integration

Worker Node:

  • kubelet - Pod lifecycle management
  • kube-proxy - Network rules
  • Container runtime - Container execution
Q: How does Kubernetes networking work?

Kubernetes networking model requirements:

  • All pods can communicate with any pod without NAT
  • All nodes can communicate with all pods without NAT
  • Pod's IP is same from inside and outside

Implementation:

  • CNI plugins (Calico, Flannel, Cilium) implement networking
  • kube-proxy manages Service to Pod routing (iptables/IPVS)
  • CoreDNS provides service discovery

Workload Questions

Q: When would you use StatefulSet vs Deployment?

Deployment - Stateless applications:

  • Web servers, APIs
  • Pods are interchangeable
  • No persistent identity needed

StatefulSet - Stateful applications:

  • Databases (MySQL, PostgreSQL)
  • Message queues (Kafka)
  • Need stable network identity (pod-0, pod-1)
  • Need stable persistent storage
  • Ordered, graceful deployment and scaling
Q: How do you implement canary deployments in Kubernetes?

Multiple approaches:

  1. Native: Two deployments with weighted Service selectors
  2. Ingress: Traffic splitting via Nginx annotations or Traefik
  3. Service Mesh: Istio/Linkerd traffic management
  4. Argo Rollouts: Progressive delivery controller
  5. Flagger: Automated canary analysis and promotion
Q: What are init containers and when would you use them?

Init containers run sequentially before main containers start. Use cases:

  • Wait for a service to be available
  • Clone a Git repo into a volume
  • Run database migrations
  • Generate config files dynamically
  • Download secrets from external vault
  • Set up file permissions on volumes
Q: Explain resource requests vs limits

Requests:

  • Guaranteed minimum resources
  • Used by scheduler for placement
  • Container can use more if available

Limits:

  • Maximum resources allowed
  • CPU: Throttled if exceeded
  • Memory: OOMKilled if exceeded

Best practice: Set requests = limits for predictable behavior (Guaranteed QoS).

Troubleshooting Scenarios

Scenario: Pod stuck in Pending state

Debugging steps:

  1. kubectl describe pod <name> - Check Events
  2. Check for "No nodes available" - Resource constraints
  3. Check for "PersistentVolumeClaim not found"
  4. Verify node selector/affinity matches nodes
  5. Check for taints without tolerations
  6. kubectl get nodes -o wide - Node status

Scenario: Pod in CrashLoopBackOff

Debugging steps:

  1. kubectl logs <pod> --previous
  2. kubectl describe pod - Check exit code
  3. Exit code 137 = OOMKilled (memory limit)
  4. Exit code 1 = Application error
  5. Check ConfigMaps/Secrets exist
  6. Verify command and args
  7. Check image tag exists

Scenario: Service not routing to pods

Debugging steps:

  1. kubectl get endpoints <service> - Check if endpoints exist
  2. Verify selector matches pod labels
  3. kubectl describe svc <service>
  4. Check if pods are Ready (readiness probe)
  5. Verify port and targetPort are correct
  6. Test with kubectl port-forward pod/<name> 8080:80

Scenario: High latency in application

Investigation:

  1. Check pod resource usage: kubectl top pods
  2. Check node resource usage: kubectl top nodes
  3. Review application logs for errors
  4. Check network policies restricting traffic
  5. Verify service mesh (if used) sidecar health
  6. Check database/external service connectivity
  7. Review CPU throttling (CFS quota)

Security Questions

Q: How do you secure a Kubernetes cluster?

Authentication & Authorization:

  • Enable RBAC, avoid cluster-admin
  • Use OIDC for user authentication
  • Principle of least privilege

Network Security:

  • Network Policies for pod isolation
  • Use TLS everywhere
  • Restrict API server access

Workload Security:

  • Pod Security Standards
  • Run as non-root
  • Read-only root filesystem
  • Scan images for vulnerabilities

Data Security:

  • Encrypt secrets at rest
  • Use external secret managers
  • Enable audit logging
Q: What are Network Policies and how do they work?

Network Policies are firewall rules for pods at L3/L4:

  • Control ingress (incoming) and egress (outgoing) traffic
  • Select pods by labels
  • Allow/deny based on namespace, pod labels, IP blocks, ports
  • Default: All traffic allowed
  • Once a policy selects a pod, all non-matched traffic is denied

Requires CNI that supports Network Policies (Calico, Cilium, Weave)

Interview Tips

Key Areas to Master

  • Pod lifecycle and debugging
  • Service types and networking
  • Deployments, rolling updates, rollbacks
  • ConfigMaps, Secrets, and environment variables
  • Persistent storage (PV, PVC, StorageClass)
  • RBAC and security contexts
  • Monitoring with Prometheus/Grafana
  • Helm basics

Pro Tips

  • Always mention that Deployment is preferred over bare ReplicaSets
  • Know the difference between liveness and readiness probes
  • Understand when to use StatefulSet vs Deployment
  • Be able to explain what happens internally when you create a pod
  • Mention security best practices proactively
  • Know common kubectl debugging commands by heart
