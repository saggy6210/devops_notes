What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform that automates deploying, scaling, and managing containerized applications. Originally developed by Google, it's now maintained by the Cloud Native Computing Foundation (CNCF).

Key Capabilities

  • Service Discovery & Load Balancing - Expose containers using DNS or IP, distribute traffic
  • Storage Orchestration - Automatically mount storage systems (local, cloud, NFS)
  • Automated Rollouts & Rollbacks - Declare desired state, K8s changes actual state
  • Self-Healing - Restarts failed containers, replaces and reschedules nodes
  • Secret & Config Management - Store and manage sensitive information
  • Horizontal Scaling - Scale applications up/down based on CPU/memory

Kubernetes Architecture

A Kubernetes cluster consists of a Control Plane (master) and one or more Worker Nodes.

Control Plane Components

kube-apiserver

The front-end for the Kubernetes control plane. All communication goes through the API server. It validates and processes REST requests, then updates etcd.

etcd

Consistent and highly-available key-value store used as Kubernetes' backing store for all cluster data. Think of it as the cluster's "database".

kube-scheduler

Watches for newly created Pods with no assigned node, and selects a node for them to run on based on resource requirements, constraints, and policies.

kube-controller-manager

Runs controller processes: Node Controller, Replication Controller, Endpoints Controller, Service Account & Token Controllers.

cloud-controller-manager

Links cluster to cloud provider's API. Manages cloud-specific controllers (Node, Route, Service Load Balancer).

Worker Node Components

kubelet

Agent running on each node. Ensures containers are running in a Pod. Takes PodSpecs and ensures described containers are healthy.

kube-proxy

Network proxy running on each node. Maintains network rules that allow communication to Pods from inside or outside the cluster.

Container Runtime

Software responsible for running containers. Kubernetes supports containerd, CRI-O, and any CRI-compliant runtime.

Key Concepts

Namespace

Virtual clusters within a physical cluster. Used to divide cluster resources between multiple users/teams. Default namespaces: default, kube-system, kube-public, kube-node-lease.

Pod

Smallest deployable unit in Kubernetes. A Pod encapsulates one or more containers, storage resources, a unique network IP, and options for how containers should run.

ReplicaSet

Ensures a specified number of pod replicas are running at any given time. Usually managed by Deployments rather than directly.

Deployment

Provides declarative updates for Pods and ReplicaSets. You describe a desired state, and the Deployment Controller changes actual state to desired state.

Service

An abstract way to expose an application running on a set of Pods as a network service. Types: ClusterIP, NodePort, LoadBalancer, ExternalName.

Interview Tip

Always remember: Pod is the smallest unit, but you rarely create Pods directly. Use Deployments for stateless apps and StatefulSets for stateful apps.

Interview Q&A

Q: What is the difference between Docker and Kubernetes?

Docker is a container runtime that packages and runs individual containers.

Kubernetes is a container orchestration platform that manages multiple containers across multiple hosts.

Think of Docker as creating the containers, and Kubernetes as managing where and how many containers run, handling scaling, networking, and self-healing.

Q: Explain the role of etcd in Kubernetes

etcd is a distributed key-value store that serves as Kubernetes' primary datastore. It stores:

  • Cluster state and configuration
  • Node information
  • Pod and service data
  • Secrets and ConfigMaps

Only the API server communicates directly with etcd. It uses the Raft consensus algorithm for high availability.

Q: What happens when a node fails in Kubernetes?

When a node fails:

  1. Node Controller detects the node is not responding (default: 40 seconds)
  2. Node is marked as NotReady
  3. After pod-eviction-timeout (default: 5 minutes), pods are evicted
  4. If pods were managed by a Deployment/ReplicaSet, new pods are scheduled on healthy nodes
  5. Standalone pods are lost unless recreated manually
Q: What is the role of kube-scheduler?

kube-scheduler assigns newly created pods to nodes. The scheduling decision considers:

  • Resource requirements - CPU, memory requests/limits
  • Hardware/software constraints - nodeSelector, affinity/anti-affinity
  • Taints and tolerations
  • Data locality - Volumes, persistent storage
  • Inter-pod affinity - Co-location requirements
Q: What is the difference between kubelet and kube-proxy?

kubelet:

  • Runs on every node
  • Ensures containers are running in pods
  • Reports node/pod status to API server
  • Executes pod lifecycle hooks

kube-proxy:

  • Runs on every node
  • Maintains network rules (iptables/IPVS)
  • Enables Service abstraction
  • Handles pod-to-pod and external-to-pod communication

Basic YAML Structure

All Kubernetes resources are defined using YAML manifests with four required fields:

Basic Pod YAML apiVersion: v1 # API version kind: Pod # Resource type metadata: # Resource metadata name: my-pod labels: app: myapp spec: # Desired state containers: - name: nginx image: nginx:latest ports: - containerPort: 80

Common Mistake

Always use specific image tags (e.g., nginx:1.21) instead of latest in production. The latest tag can lead to unexpected changes during pod restarts.
