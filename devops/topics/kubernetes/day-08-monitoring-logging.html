Kubernetes Monitoring

Effective monitoring covers multiple layers: cluster health, node metrics, pod/container metrics, and application metrics.

Prometheus

Time-series database and monitoring system. Pull-based metrics collection with powerful query language (PromQL).

Grafana

Visualization and dashboarding. Supports multiple data sources including Prometheus.

Alertmanager

Handles alerts from Prometheus. Supports grouping, silencing, and routing to various receivers.

Metrics Server

Cluster-wide resource metrics. Required for HPA. Lightweight alternative to Heapster.

Metrics Server

Metrics Server collects resource metrics from kubelets and exposes them via the Metrics API for use by HPA and kubectl top.

Install Metrics Server kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml # Verify installation kubectl get deployment metrics-server -n kube-system # Use metrics kubectl top nodes kubectl top pods kubectl top pods -A --containers

Prometheus Stack

Install kube-prometheus-stack # Add Helm repo helm repo add prometheus-community https://prometheus-community.github.io/helm-charts helm repo update # Install with Grafana, Alertmanager, and exporters helm install prometheus prometheus-community/kube-prometheus-stack \ --namespace monitoring \ --create-namespace \ --set grafana.adminPassword=admin123 # Access Grafana kubectl port-forward svc/prometheus-grafana 3000:80 -n monitoring

Key Components

  • Prometheus Server - Scrapes and stores metrics
  • Node Exporter - Hardware/OS metrics from nodes
  • kube-state-metrics - K8s object state metrics
  • Alertmanager - Alert handling and routing
  • Grafana - Dashboards and visualization

ServiceMonitor

ServiceMonitor is a CRD that defines how to scrape metrics from Services.

ServiceMonitor Example apiVersion: monitoring.coreos.com/v1 kind: ServiceMonitor metadata: name: myapp-monitor labels: release: prometheus # Must match Prometheus selector spec: selector: matchLabels: app: myapp endpoints: - port: metrics interval: 30s path: /metrics namespaceSelector: matchNames: - default

Best Practice

Ensure your ServiceMonitor has labels that match the Prometheus operator's serviceMonitorSelector. Check with: kubectl get prometheus -o yaml

Kubernetes Logging

Kubernetes doesn't provide native log aggregation. Common patterns include node-level logging agents and sidecar containers.

Logging Architecture

  • Node-level agent - DaemonSet (Fluentd, Fluent Bit) on each node
  • Sidecar container - Log shipping container per pod
  • Direct push - Application sends logs directly to backend

EFK Stack

Elasticsearch + Fluentd + Kibana. Traditional, feature-rich log aggregation.

Loki + Grafana

Lightweight, Prometheus-inspired logging. Labels-based indexing. Cost-effective.

Log Commands

kubectl logs # Basic logs kubectl logs <pod-name> kubectl logs <pod-name> -c <container> # Specific container kubectl logs -f <pod-name> # Stream/follow kubectl logs --previous <pod-name> # Previous container # Multiple pods kubectl logs -l app=nginx # By label kubectl logs -l app=nginx --all-containers # Time-based kubectl logs --since=1h <pod-name> kubectl logs --since-time=2024-01-01T00:00:00Z <pod-name> kubectl logs --tail=100 <pod-name> # Last 100 lines
Loki Stack Installation # Add Grafana repo helm repo add grafana https://grafana.github.io/helm-charts # Install Loki stack helm install loki grafana/loki-stack \ --namespace logging \ --create-namespace \ --set grafana.enabled=true \ --set promtail.enabled=true # Access Grafana (add Loki as data source) kubectl port-forward svc/loki-grafana 3000:80 -n logging

Alerting

PrometheusRule Example apiVersion: monitoring.coreos.com/v1 kind: PrometheusRule metadata: name: pod-alerts labels: release: prometheus spec: groups: - name: pod.rules rules: - alert: PodCrashLooping expr: rate(kube_pod_container_status_restarts_total[15m]) > 0 for: 5m labels: severity: warning annotations: summary: "Pod {{ $labels.pod }} is crash looping" description: "Pod {{ $labels.pod }} in {{ $labels.namespace }} has restarted more than once in 15 minutes." - alert: HighMemoryUsage expr: container_memory_usage_bytes / container_spec_memory_limit_bytes > 0.9 for: 5m labels: severity: critical annotations: summary: "High memory usage detected"

Interview Q&A

Q: What is the difference between Metrics Server and Prometheus?

Metrics Server:

  • Lightweight, cluster-wide resource metrics
  • Short-term memory (no storage)
  • Used by HPA, VPA, kubectl top
  • Only CPU/memory metrics

Prometheus:

  • Full monitoring system with time-series DB
  • Long-term storage
  • Rich query language (PromQL)
  • Custom metrics, alerting, visualization
Q: How does Prometheus discover targets in Kubernetes?

Prometheus uses service discovery mechanisms:

  • kubernetes_sd_config - Native K8s discovery for nodes, pods, services, endpoints
  • ServiceMonitor - CRD managed by Prometheus Operator
  • PodMonitor - Directly target pods without services
  • Annotations - prometheus.io/scrape, prometheus.io/port
Q: What logging solution would you recommend for Kubernetes?

Depends on requirements:

  • Loki + Grafana: Cost-effective, integrates with Prometheus, good for most use cases
  • EFK/ELK: Powerful full-text search, mature ecosystem, higher resource usage
  • Cloud-native: AWS CloudWatch, GCP Stackdriver, Azure Monitor - if already in cloud
  • DataDog/Splunk: Enterprise features, SaaS options
Q: How do you troubleshoot a pod that's not producing logs?
  1. Check pod status: kubectl describe pod
  2. Check if container is running: kubectl get pod -o wide
  3. Try kubectl logs --previous for crashed containers
  4. Exec into pod: kubectl exec -it pod -- sh
  5. Check application log path inside container
  6. Verify logging agent (Fluentd/Promtail) is running
  7. Check node-level logs: /var/log/containers/
