RBAC Overview

Role-Based Access Control (RBAC) regulates access to Kubernetes resources based on the roles of individual users or service accounts.

Role

Defines permissions within a namespace

ClusterRole

Defines cluster-wide permissions

RoleBinding

Binds Role to users in a namespace

ClusterRoleBinding

Binds ClusterRole cluster-wide

Roles & ClusterRoles

Role (Namespace-scoped) apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: namespace: dev name: pod-reader rules: - apiGroups: [""] # "" = core API group resources: ["pods"] verbs: ["get", "list", "watch"] - apiGroups: [""] resources: ["pods/log"] verbs: ["get"]
ClusterRole (Cluster-wide) apiVersion: rbac.authorization.k8s.io/v1 kind: ClusterRole metadata: name: secret-reader rules: - apiGroups: [""] resources: ["secrets"] verbs: ["get", "list"] - apiGroups: ["apps"] resources: ["deployments"] verbs: ["get", "list", "create", "update", "delete"]

Common Verbs

  • get, list, watch - Read operations
  • create, update, patch - Write operations
  • delete, deletecollection - Delete operations
  • * - All verbs (use cautiously)

RoleBindings

RoleBinding apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: read-pods namespace: dev subjects: - kind: User name: jane apiGroup: rbac.authorization.k8s.io - kind: ServiceAccount name: my-app namespace: dev roleRef: kind: Role name: pod-reader apiGroup: rbac.authorization.k8s.io
ClusterRoleBinding apiVersion: rbac.authorization.k8s.io/v1 kind: ClusterRoleBinding metadata: name: admin-binding subjects: - kind: Group name: admins apiGroup: rbac.authorization.k8s.io roleRef: kind: ClusterRole name: cluster-admin apiGroup: rbac.authorization.k8s.io

ServiceAccounts

ServiceAccounts provide identity for processes running in pods. Each namespace has a default SA.

ServiceAccount apiVersion: v1 kind: ServiceAccount metadata: name: my-app-sa namespace: dev --- # Pod using ServiceAccount apiVersion: v1 kind: Pod metadata: name: my-app spec: serviceAccountName: my-app-sa automountServiceAccountToken: false # Security: disable if not needed containers: - name: app image: myapp:latest
ServiceAccount Commands kubectl create serviceaccount my-sa kubectl get serviceaccounts kubectl describe sa my-sa # Create token (K8s 1.24+) kubectl create token my-sa --duration=1h

Pod Security

SecurityContext apiVersion: v1 kind: Pod metadata: name: secure-pod spec: securityContext: runAsUser: 1000 runAsGroup: 3000 fsGroup: 2000 runAsNonRoot: true containers: - name: app image: myapp:latest securityContext: allowPrivilegeEscalation: false readOnlyRootFilesystem: true capabilities: drop: - ALL add: - NET_BIND_SERVICE

Pod Security Standards (PSS)

  • Privileged - Unrestricted policy (avoid in production)
  • Baseline - Minimally restrictive, prevents known privilege escalations
  • Restricted - Heavily restricted, follows security best practices
Namespace Pod Security Labels apiVersion: v1 kind: Namespace metadata: name: secure-ns labels: pod-security.kubernetes.io/enforce: restricted pod-security.kubernetes.io/warn: restricted pod-security.kubernetes.io/audit: restricted

RBAC Commands

RBAC Management # Check permissions kubectl auth can-i create pods kubectl auth can-i delete deployments --as=jane kubectl auth can-i '*' '*' --as=system:serviceaccount:dev:my-sa # Create role/binding imperatively kubectl create role pod-reader --verb=get,list,watch --resource=pods kubectl create rolebinding read-pods --role=pod-reader --user=jane # View RBAC kubectl get roles,rolebindings -n dev kubectl get clusterroles,clusterrolebindings kubectl describe role pod-reader

Debugging Tip

Use kubectl auth can-i --list to see all allowed actions for current user, or --as=username to impersonate.

Security Best Practices

Critical Security Rules

  • Never grant cluster-admin to applications
  • Use namespace-scoped roles when possible
  • Avoid wildcards (*) in rules
  • Don't run containers as root
  • Enable audit logging
  • Rotate secrets and tokens regularly

Recommended Practices

  • Apply principle of least privilege
  • Use separate ServiceAccounts per application
  • Disable automountServiceAccountToken when not needed
  • Use Network Policies to restrict pod communication
  • Scan images for vulnerabilities
  • Enable Pod Security Standards

Interview Q&A

Q: What is the difference between Role and ClusterRole?

Role: Namespace-scoped. Defines permissions within a specific namespace.

ClusterRole: Cluster-scoped. Can define permissions for:

  • Cluster-scoped resources (nodes, PVs)
  • Non-resource endpoints (/healthz)
  • Namespaced resources across all namespaces
Q: How do you grant a ServiceAccount access to the API?
  1. Create a ServiceAccount
  2. Create a Role/ClusterRole with needed permissions
  3. Bind the Role to the ServiceAccount with RoleBinding/ClusterRoleBinding
  4. Assign ServiceAccount to Pod via serviceAccountName
Q: What are the three API groups you commonly use in RBAC rules?
  • "" (empty string) - Core API group (pods, services, secrets, etc.)
  • "apps" - Deployments, StatefulSets, DaemonSets, ReplicaSets
  • "rbac.authorization.k8s.io" - Roles, ClusterRoles, Bindings

Others: batch (jobs), networking.k8s.io (ingress), storage.k8s.io (PV, StorageClass)

Q: How do you prevent containers from running as root?
  1. Set runAsNonRoot: true in securityContext
  2. Specify runAsUser with non-zero UID
  3. Apply Pod Security Standards with "restricted" profile
  4. Use admission controllers (OPA/Gatekeeper, Kyverno)
