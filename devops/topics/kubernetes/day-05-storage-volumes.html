Volume Types

Kubernetes supports various volume types for different use cases.

emptyDir

Temporary directory created when Pod is assigned to a node. Deleted when Pod is removed. Good for cache, scratch space, sharing between containers.

hostPath

Mounts a file or directory from the host node. Data persists across pod restarts on same node. Use cautiously - creates node dependency.

PersistentVolume (PV)

Cluster-wide storage resource provisioned by admin or dynamically. Exists independently of pods. Supports NFS, iSCSI, cloud storage.

PersistentVolumeClaim (PVC)

Request for storage by a user. Claims are bound to PVs that satisfy size and access mode requirements.

Basic Volume Examples

emptyDir Volume apiVersion: v1 kind: Pod metadata: name: cache-pod spec: containers: - name: app image: nginx volumeMounts: - name: cache-volume mountPath: /cache - name: sidecar image: busybox volumeMounts: - name: cache-volume mountPath: /data volumes: - name: cache-volume emptyDir: sizeLimit: 500Mi # Optional limit
hostPath Volume apiVersion: v1 kind: Pod metadata: name: hostpath-pod spec: containers: - name: app image: nginx volumeMounts: - name: host-data mountPath: /data volumes: - name: host-data hostPath: path: /var/data type: DirectoryOrCreate # Directory, File, etc.

hostPath Warning

Avoid hostPath in production. It creates node affinity, potential security risks, and doesn't work across nodes.

PersistentVolumes & Claims

Access Modes

  • ReadWriteOnce (RWO) - Volume can be mounted read-write by single node
  • ReadOnlyMany (ROX) - Volume can be mounted read-only by many nodes
  • ReadWriteMany (RWX) - Volume can be mounted read-write by many nodes
  • ReadWriteOncePod (RWOP) - Volume mounted read-write by single pod (K8s 1.22+)
PersistentVolume apiVersion: v1 kind: PersistentVolume metadata: name: pv-nfs spec: capacity: storage: 10Gi accessModes: - ReadWriteMany persistentVolumeReclaimPolicy: Retain storageClassName: nfs nfs: server: nfs-server.example.com path: /exports/data
PersistentVolumeClaim apiVersion: v1 kind: PersistentVolumeClaim metadata: name: my-pvc spec: accessModes: - ReadWriteOnce resources: requests: storage: 5Gi storageClassName: standard # Must match PV or StorageClass
Pod Using PVC apiVersion: v1 kind: Pod metadata: name: app-with-storage spec: containers: - name: app image: nginx volumeMounts: - name: data mountPath: /usr/share/nginx/html volumes: - name: data persistentVolumeClaim: claimName: my-pvc

StorageClasses

StorageClasses enable dynamic provisioning of PersistentVolumes. Admins define storage "classes" with different performance characteristics.

StorageClass Example (AWS EBS) apiVersion: storage.k8s.io/v1 kind: StorageClass metadata: name: fast-ssd provisioner: ebs.csi.aws.com parameters: type: gp3 iops: "3000" throughput: "125" reclaimPolicy: Delete volumeBindingMode: WaitForFirstConsumer allowVolumeExpansion: true
PVC with StorageClass apiVersion: v1 kind: PersistentVolumeClaim metadata: name: fast-pvc spec: accessModes: - ReadWriteOnce storageClassName: fast-ssd resources: requests: storage: 20Gi

Reclaim Policies

  • Retain - Manual reclamation. PV becomes "Released" after PVC deletion.
  • Delete - Automatically delete the underlying storage (default for dynamic).
  • Recycle - Deprecated. Basic scrub (rm -rf).

StatefulSet with Storage

StatefulSet with VolumeClaimTemplate apiVersion: apps/v1 kind: StatefulSet metadata: name: mysql spec: serviceName: mysql replicas: 3 selector: matchLabels: app: mysql template: metadata: labels: app: mysql spec: containers: - name: mysql image: mysql:8.0 volumeMounts: - name: data mountPath: /var/lib/mysql volumeClaimTemplates: - metadata: name: data spec: accessModes: ["ReadWriteOnce"] storageClassName: fast-ssd resources: requests: storage: 10Gi

StatefulSet Storage

VolumeClaimTemplates create unique PVCs for each pod (data-mysql-0, data-mysql-1, etc.). PVCs persist even when pods are deleted.

Storage Commands

PV/PVC Commands kubectl get pv kubectl get pvc kubectl describe pv pv-name kubectl describe pvc pvc-name kubectl get storageclass # Expand PVC (if allowed) kubectl patch pvc my-pvc -p '{"spec":{"resources":{"requests":{"storage":"10Gi"}}}}'

Interview Q&A

Q: What is the difference between PV and PVC?

PersistentVolume (PV): Cluster-level storage resource provisioned by admin or dynamically. Represents actual storage.

PersistentVolumeClaim (PVC): User's request for storage. Acts as a "claim" that binds to a suitable PV.

Analogy: PV is like a disk in a data center, PVC is a request form to use that disk.

Q: What is static vs dynamic provisioning?

Static: Admin manually creates PVs. PVCs bind to existing PVs that match requirements.

Dynamic: PVs are automatically created when PVC requests storage. Requires StorageClass with provisioner.

Dynamic is preferred as it reduces admin overhead and enables self-service.

Q: What happens when a PVC is deleted?

Depends on the reclaimPolicy:

  • Delete: PV and underlying storage are deleted
  • Retain: PV becomes "Released", data preserved, manual cleanup needed

A Released PV cannot be bound to new PVC until manually reclaimed or data deleted.

Q: When would you use emptyDir vs PVC?

emptyDir:

  • Temporary storage (cache, scratch space)
  • Sharing data between containers in same pod
  • Data that can be lost when pod restarts

PVC:

  • Persistent data (databases, uploads)
  • Data that must survive pod restarts
  • Stateful applications
Q: How do you resize a PVC?
  1. StorageClass must have allowVolumeExpansion: true
  2. Edit PVC to increase storage request
  3. For file systems, pod may need restart for resize to take effect

Note: PVCs can only be expanded, not shrunk.
